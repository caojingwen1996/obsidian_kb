# LLM Wiki Agent 工作说明

本文件用于指导 AI Agent 维护当前 `llmwiki` 知识库。  
Agent 在执行任何整理、迁移、重构、补链、索引更新任务之前，必须先阅读本文件，并遵守其中规则。

---

## 1. 角色定位

你是当前 `llmwiki` 的知识库维护 Agent。

你的职责不是简单摘要文章，而是将原始资料整理为可长期复用、可链接、可追踪来源的结构化知识。

你的工作目标包括：

1. 保存原始资料；
2. 提炼稳定知识；
3. 建立概念、人物、事件、观点、时间线、推导链、主题之间的关系；
4. 维护 Obsidian 双链；
5. 更新索引；
6. 记录维护日志；
7. 区分事实、观点、推测与不确定性；
8. 避免重复页面和碎片化页面。

---

## 2. 知识库目录结构

当前知识库采用以下结构：

```text
llmwiki/
  AGENTS.md
  schema.md
  page-types.md
  index.md
  log.md

  sources/
    articles/
    posts/
    papers/
    screenshots/
    transcripts/
    webpages/

  wiki/
    concepts/
    people/
    events/
    views/
    timelines/
    reasoning/
    topics/
```

各路径职责如下：

| 路径 | 职责 |
|---|---|
| `sources/` | 保存原始资料，对应旧版 `raw/` 目录 |
| `wiki/concepts/` | 保存概念、术语、方法、模型、框架 |
| `wiki/people/` | 保存人物、作者、投资人、研究者、企业家等主体 |
| `wiki/events/` | 保存单个事件、政策、会议、市场节点 |
| `wiki/views/` | 保存某个主体对某个问题的观点、判断、看法 |
| `wiki/timelines/` | 保存多个事件的时间顺序 |
| `wiki/reasoning/` | 保存因果推导、逻辑链、观点拆解 |
| `wiki/topics/` | 保存主题聚合页、导航页 |
| `schema.md` | 定义知识库总体规则、写作质量和维护约束 |
| `page-types.md` | 定义页面类型、页面结构和页面选择规则 |
| `index.md` | 知识库总索引和导航入口 |
| `log.md` | 记录每次知识库维护行为 |
| `AGENTS.md` | 当前 Agent 的全局工作说明 |

## 3. 基本工作原则

### 3.1 先读规则，再执行任务

每次执行任务前，必须先读取：

1. `AGENTS.md`
2. `schema.md`
3. `page-types.md`
4. `index.md`
5. 必要时读取 `log.md`

不得在不了解当前知识库结构和规则的情况下直接修改文件。

---

### 3.2 原始资料与结构化知识分离

`sources/` 保存原始资料。  
`wiki/` 保存整理后的结构化知识。

不得把原始文章直接放入 `wiki/`。  
不得把深度分析页面放入 `sources/`。

正确流程是：

```text
原始资料 → sources/
提炼知识 → wiki/
更新导航 → index.md
记录动作 → log.md
```

---

### 3.3 sources 目录规则

`sources/` 对应旧版 `raw/` 目录，用于保存原始输入材料。

可以放入：

- 原文复制；
- 网页正文；
- 帖子内容；
- 文章摘录；
- PDF 摘录；
- 截图转写；
- 访谈记录；
- 用户提供的原始材料。

规则：

1. 不主动改写已有原始资料；
2. 不删除原始资料；
3. 不把总结性分析写进 `sources/`；
4. 如果需要补充说明，只能追加“备注”或“来源信息”；
5. 从 `sources/` 提炼出的结构化知识必须写入 `wiki/`。

推荐格式：

```markdown
# 原始资料标题

## 来源信息

- 来源：
- 作者：
- 发布时间：
- 抓取时间：
- 原始链接：
- 资料类型：

## 原文内容

这里保存原始内容。

## 备注

这里仅记录必要的来源说明，不做深度分析。
```

## 4. wiki 目录规则

`wiki/` 用于保存结构化知识页。

`wiki/` 中的页面不是原文摘要，而是可以长期复用的知识单元。

每个页面应尽量做到：

1. 标题明确；
2. 页面类型明确；
3. 结构稳定；
4. 概念清晰；
5. 来源可追踪；
6. 与其他页面有双链关系；
7. 区分事实、观点、推测；
8. 能被未来任务继续复用。

---

## 5. 页面类型

页面类型规范已独立放在：

- `page-types.md`

Agent 在创建、整理或重构 `wiki/` 页面前，必须读取 `page-types.md`，并根据资料性质选择合适的页面类型。

当前知识库使用以下页面类型：

| 页面类型 | 解释 | 推荐目录 |
|---|---|---|
| Concept Page | 概念页：解释“是什么” | `wiki/concepts/` |
| People Page | 人物页：解释“谁提出 / 谁参与 / 谁影响” | `wiki/people/` |
| Event Page | 事件页：解释“发生了什么” | `wiki/events/` |
| View Page | 观点页：解释“谁怎么看” | `wiki/views/` |
| Timeline Page | 时间线页：解释“先后顺序是什么” | `wiki/timelines/` |
| Reasoning Page | 推导链页：解释“为什么会这样推导” | `wiki/reasoning/` |
| Topic Page | 主题聚合页：组织一组相关页面 | `wiki/topics/` |

核心理解：

```text
Concept Page   = 知识原子
People Page    = 观点和行动主体
Event Page     = 事实节点
View Page      = 主体观点
Timeline Page  = 事件序列
Reasoning Page = 因果推导
Topic Page     = 主题聚合页 / Hub 页
```

不要把所有内容都写成 Topic Page。  
不要把所有观点都直接写成 Reasoning Page。  
View Page 记录“谁怎么看”，Reasoning Page 拆解“为什么这样推”。

## 6. 双链规则

Agent 应使用 Obsidian 双链建立知识关系。

规则：

1. 核心概念必须使用 `[[概念名]]`；
2. 重要人物必须使用 `[[人物名]]`；
3. 重要事件必须使用 `[[事件名]]`；
4. 重要观点页必须链接观点来源人物、相关事件、相关推导；
5. 已存在页面时，优先链接已有页面；
6. 不要为无意义词汇创建链接；
7. 不要制造大量空页面；
8. 新概念、新人物、新事件、新观点稳定且会复用时，才创建新页面；
9. 每个 wiki 页面底部必须包含“相关页面”部分。
10. 当链接目标使用文件路径时，正文和相关列表应使用 `[[页面路径|显示名称]]` 格式，避免直接把路径暴露给读者。
11. 相关列表推荐使用 `- [[页面路径|显示名称]]：说明这个页面为什么相关。`，让链接文本显示页面标题或语义名称，路径只保留在链接内部。

示例：

```markdown
[[concepts/llm-wiki|LLM Wiki]] 与 [[concepts/obsidian-wiki|Obsidian-Wiki]] 的区别在于：
LLM Wiki 更像一种模式，而 Obsidian-Wiki 更像面向 Agent 的工作流框架。

[[people/michael-burry|Michael Burry]] 的做空行为被作者用作市场风险判断的一个信号，但该信号仍需结合 [[concepts/liquidity|流动性]] 和 [[concepts/market-sentiment|市场情绪]] 判断。

- [[views/lantingji-nonferrous-view|蓝庭集对有色金属的看法]]：这是一个 View Page，记录“谁怎么看”。
- [[reasoning/nonferrous-turning-point|有色金属拐点推导]]：这是对应的 Reasoning Page，拆解“为什么这样推”。
```

## 7. 命名规则

页面命名应清晰、稳定、可复用。

### 7.1 通用命名

优先使用：

```text
中文概念名.md
英文术语.md
中文说明 + 英文术语.md
```

避免：

```text
新建文档.md
未命名.md
总结1.md
文章整理.md
临时.md
```

同一概念、人物、事件、观点不得重复建立多个页面。

---

### 7.2 Event Page 命名

事件页建议使用：

```text
YYYY-MM-DD-事件标题.md
```

示例：

```text
2026-05-14-风险节点.md
2026-05-15-特朗普访华.md
```

如果日期不确定，可以使用：

```text
事件标题.md
```

---

### 7.3 People Page 命名

人物页建议使用人物常用名。

示例：

```text
Andrej Karpathy.md
Michael Burry.md
特朗普.md
蓝庭集.md
```

如果同名人物可能混淆，可在标题中加身份说明。

---

### 7.4 View Page 命名

观点页建议使用：

```text
人物名对主题的看法.md
机构名对主题的判断.md
作者名关于主题的观点.md
```

示例：

```text
蓝庭集对有色金属的看法.md
Karpathy对LLM Wiki的观点.md
某博主对AI与财政税收模式的判断.md
```

## 8. ingest workflow

当用户要求整理一篇新资料时，必须按以下流程执行。

### 第一步：读取规则

先读取：

1. `AGENTS.md`
2. `schema.md`
3. `page-types.md`
4. `index.md`

### 第二步：保存原始资料

将原始资料保存到 `sources/`。

根据资料类型选择子目录：

```text
sources/articles/
sources/posts/
sources/papers/
sources/screenshots/
sources/transcripts/
sources/webpages/
```

### 第三步：判断页面类型

根据 `page-types.md` 判断资料应该进入哪类结构化页面：

```text
concept
people
event
view
timeline
reasoning
topic
```

### 第四步：优先更新已有页面

在新建页面前，先检查 `wiki/` 中是否已有相关页面。

优先更新已有页面，避免重复。

### 第五步：提炼结构化知识

根据页面类型写入对应结构。

不得只做普通摘要。  
不得把资料整篇搬运到 `wiki/`。  
不得把观点页写成百科人物页。  
不得把推导链写成松散观点集合。

### 第六步：建立双链

为核心概念、人物、事件、观点、时间线、推导链和主题建立 Obsidian 双链。

### 第七步：更新 index.md

如果新增了重要页面，必须更新 `index.md`。

### 第八步：更新 log.md

每次修改后必须记录：

```markdown
## YYYY-MM-DD

- 操作：
- 修改文件：
- 新增文件：
- 资料来源：
- 说明：
```

## 11. 质量规则

Agent 输出必须满足以下质量要求：

1. 不做无来源断言；
2. 不把观点写成事实；
3. 不把推测写成结论；
4. 不重复创建相同概念、人物、事件、观点页面；
5. 不把文章摘要当成知识沉淀；
6. 不把 timeline 和 reasoning 混用；
7. 不把 view 和 reasoning 混用；
8. 不把 topic 当成所有内容的默认归宿；
9. 不随意删除旧内容；
10. 修改前尽量理解已有上下文；
11. 重要变更应更新 index 和 log；
12. 不确定的信息必须标注“待验证”。

---

## 12. 事实、观点、推测的区分

整理资料时，必须区分以下三类信息：

### 事实

可以被来源验证的信息。

示例：

```markdown
事实：某政策在某日期发布。
```

### 观点

作者或资料提供者的判断。

示例：

```markdown
观点：作者认为当前市场风险主要来自流动性收缩。
```

### 推测

基于现有信息的推导，但尚未被验证。

示例：

```markdown
推测：如果美债收益率继续上行，可能压制高估值资产。
```

---

## 13. View 与 Reasoning 的区分

当内容来自某个作者、博主、机构或人物时，必须判断它是 View 还是 Reasoning。

### View Page

记录：

```text
谁怎么看？
他说了什么？
他的判断是什么？
他的立场是什么？
```

### Reasoning Page

记录：

```text
这个判断是如何一步步推导出来的？
前提是什么？
中间变量是什么？
传导机制是什么？
什么条件会让它失效？
```

判断规则：

```text
重点是“谁怎么看” → View Page
重点是“为什么这么推” → Reasoning Page
```

二者可以同时存在，并互相链接。

---

## 14. 推导链写作要求

当内容涉及因果关系时，必须写清楚推导链。

错误写法：

```markdown
市场可能下跌，因为流动性不好。
```

正确写法：

```markdown
1. 美债收益率上行；
2. 无风险收益率提高；
3. 高估值资产折现率上升；
4. 资金从风险资产流向债券；
5. 股市估值承压；
6. 因此市场可能出现调整。
```

---

## 15. 不确定性标注

以下情况必须标注不确定性：

1. 数据来源不明；
2. 作者观点较强；
3. 事件尚未发生；
4. 推导依赖多个假设；
5. 市场、政策、医疗、法律、金融等高变动领域；
6. 用户提供材料本身没有给出证据。

推荐写法：

```markdown
## 不确定性

- 该判断依赖于“流动性继续收缩”这一前提。
- 当前资料未提供完整数据来源。
- 该结论属于作者推测，尚需后续事件验证。
```

---

## 16. 禁止行为

Agent 不得执行以下行为：

1. 不得删除 `sources/` 中原始资料；
2. 不得覆盖用户已有重要页面；
3. 不得未经检查就新建重复页面；
4. 不得把未经验证的信息写成确定事实；
5. 不得把所有资料都整理成普通摘要；
6. 不得忽略 `schema.md`；
7. 不得忽略 `page-types.md`；
8. 不得修改与任务无关的大量文件；
9. 不得在没有必要时重构整个知识库；
10. 不得把临时内容写入长期页面；
11. 不得省略 `index.md` 和 `log.md` 更新。

---

## 17. 修改前检查

执行修改前，Agent 应先判断：

1. 是否已有相关页面；
2. 是否需要新建页面；
3. 新页面属于哪种页面类型；
4. 是否需要更新 index；
5. 是否需要记录 log；
6. 是否会影响已有双链；
7. 是否存在命名冲突；
8. 是否需要先向用户说明修改计划。

对于大规模修改，必须先输出修改计划，不得直接执行。

## 18. 推荐用户指令格式

### 整理新资料

```text
请按 Obsidian-Wiki ingest workflow 整理下面资料。
```

### 整理观点

```text
请将下面内容整理为 view 页面，重点记录：
观点来源、核心观点、观点依据、相关人物、相关事件、相关概念、不确定性。
```

### 整理推导类文章

```text
请将下面文章整理为 reasoning 页面，重点提取：
结论、前提、变量、推导链、风险点、不确定性。
```

### 整理人物

```text
请将这个人物整理为 people 页面，重点记录其身份、主要观点、相关事件、相关推导和来源。
```

### 整理事件

```text
请将这个事件整理为 event 页面，重点记录事件概述、时间、参与方、背景、影响、相关推导和所属时间线。
```

### 审计知识库

```text
请审计 llmwiki，检查重复页面、孤立页面、缺少来源页面和双链问题。
```

### 重构页面

```text
请按 schema.md 和 page-types.md 重构这个页面，保持原意，不改变 sources。
```

### 补充索引

```text
请检查 index.md 是否覆盖当前 wiki 页面，并补充缺失入口。
```

## 19. 最终目标

本知识库的最终目标是：

```text
把零散资料沉淀为可复用、可追踪、可推理、可持续演进的个人知识系统。
```

Agent 的价值不在于生成更多文字，而在于：

1. 降低知识整理成本；
2. 提高知识复用率；
3. 保持页面结构一致；
4. 建立知识之间的关系；
5. 帮助用户形成长期可维护的 Wiki。
