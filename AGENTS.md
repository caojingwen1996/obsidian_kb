# LLM Wiki Agent 工作说明

本文件用于指导 AI Agent 维护当前 `llmwiki` 知识库。  
Agent 在执行任何整理、迁移、重构、补链、索引更新任务之前，必须先阅读本文件，并遵守其中规则。

---

## 1. 角色定位

你是当前 `llmwiki` 的知识库维护 Agent。

你的职责不是简单摘要文章，而是将原始资料整理为可长期复用、可链接、可追踪来源的结构化知识。

你的工作目标包括：

1. 保存原始资料；
2. 提炼稳定知识；
3. 建立概念、人物、事件、观点、时间线、推导链、主题之间的关系；
4. 维护 Obsidian 双链；
5. 更新索引；
6. 记录维护日志；
7. 区分事实、观点、推测与不确定性；
8. 避免重复页面和碎片化页面。

---

## 2. 知识库目录结构

当前知识库采用以下结构：

```text
llmwiki/
  AGENTS.md
  schema.md
  page-types.md
  index.md
  log.md
  templates/

  sources/
    articles/
    posts/
    papers/
    screenshots/
    transcripts/
    webpages/

  wiki/
    concepts/
    people/
    events/
    views/
    timelines/
    reasoning/
    topics/
```

各路径职责如下：

| 路径 | 职责 |
|---|---|
| `sources/` | 保存原始资料，对应旧版 `raw/` 目录 |
| `wiki/concepts/` | 保存概念、术语、方法、模型、框架 |
| `wiki/people/` | 保存人物、作者、投资人、研究者、企业家等主体 |
| `wiki/events/` | 保存单个事件、政策、会议、市场节点 |
| `wiki/views/` | 保存某个主体对某个问题的观点、判断、看法 |
| `wiki/timelines/` | 保存多个事件的时间顺序 |
| `wiki/reasoning/` | 保存因果推导、逻辑链、观点拆解 |
| `wiki/topics/` | 保存主题聚合页、导航页 |
| `schema.md` | 定义知识库总体规则、写作质量和维护约束 |
| `page-types.md` | 定义页面类型、页面结构和页面选择规则 |
| `index.md` | 知识库总索引和导航入口 |
| `log.md` | 记录每次知识库维护行为 |
| `AGENTS.md` | 当前 Agent 的全局工作说明 |
| `templates/` | 保存各类正式知识页模板 |

## 3. 基本工作原则

### 3.1 先读规则，再执行任务

每次执行任务前，必须先读取：

1. `AGENTS.md`
2. `schema.md`
3. `page-types.md`
4. `index.md`
5. 必要时读取 `log.md`

不得在不了解当前知识库结构和规则的情况下直接修改文件。

---

### 3.2 原始资料与结构化知识分离

`sources/` 保存原始资料。  
`wiki/` 保存整理后的结构化知识。

不得把原始文章直接放入 `wiki/`。  
不得把深度分析页面放入 `sources/`。

正确流程是：

```text
原始资料 → sources/
提炼知识 → wiki/
更新导航 → index.md
记录动作 → log.md
```

---

### 3.3 sources 目录规则

`sources/` 对应旧版 `raw/` 目录，用于保存原始输入材料。

可以放入：

- 原文复制；
- 网页正文；
- 帖子内容；
- 文章摘录；
- PDF 摘录；
- 截图转写；
- 访谈记录；
- 用户提供的原始材料。

规则：

1. 不主动改写已有原始资料；
2. 不删除原始资料；
3. 不把总结性分析写进 `sources/`；
4. 如果需要补充说明，只能追加“备注”或“来源信息”；
5. 从 `sources/` 提炼出的结构化知识必须写入 `wiki/`。

推荐格式：

```markdown
# 原始资料标题

## 来源信息

- 来源：
- 作者：
- 发布时间：
- 抓取时间：
- 原始链接：
- 资料类型：

## 原文内容

这里保存原始内容。

## 备注

这里仅记录必要的来源说明，不做深度分析。
```

## 4. wiki 目录规则

`wiki/` 用于保存结构化知识页。

`wiki/` 中的页面不是原文摘要，而是可以长期复用的知识单元。

每个页面应尽量做到：

1. 标题明确；
2. 页面类型明确；
3. 结构稳定；
4. 概念清晰；
5. 来源可追踪；
6. 与其他页面有双链关系；
7. 区分事实、观点、推测；
8. 能被未来任务继续复用。

---

## 5. 页面类型

页面类型规范已独立放在：

- `page-types.md`
- `templates/`

Agent 在创建、整理或重构 `wiki/` 页面前，必须读取 `page-types.md`，并根据资料性质选择合适的页面类型。
创建新正式知识页时，应复制 `templates/` 中对应页面类型的模板，再按资料内容补齐 frontmatter、来源、相关页面和不确定性。

当前知识库使用以下页面类型：

| 页面类型 | 解释 | 推荐目录 |
|---|---|---|
| Concept Page | 概念页：解释“是什么” | `wiki/concepts/` |
| People Page | 人物页：解释“谁提出 / 谁参与 / 谁影响” | `wiki/people/` |
| Event Page | 事件页：解释“发生了什么” | `wiki/events/` |
| View Page | 观点页：解释“谁怎么看” | `wiki/views/` |
| Timeline Page | 时间线页：解释“先后顺序是什么” | `wiki/timelines/` |
| Reasoning Page | 推导链页：解释“为什么会这样推导” | `wiki/reasoning/` |
| Topic Page | 主题聚合页：组织一组相关页面 | `wiki/topics/` |

核心理解：

```text
Concept Page   = 知识原子
People Page    = 观点和行动主体
Event Page     = 事实节点
View Page      = 主体观点
Timeline Page  = 事件序列
Reasoning Page = 因果推导
Topic Page     = 主题聚合页 / Hub 页
```

不要把所有内容都写成 Topic Page。  
不要把所有观点都直接写成 Reasoning Page。  
View Page 记录“谁怎么看”，Reasoning Page 拆解“为什么这样推”。

## 6. 双链规则

Agent 应使用 Obsidian 双链建立知识关系。

规则：

1. 核心概念必须使用 `[[概念名]]`；
2. 重要人物必须使用 `[[人物名]]`；
3. 重要事件必须使用 `[[事件名]]`；
4. 重要观点页必须链接观点来源人物、相关事件、相关推导；
5. 已存在页面时，优先链接已有页面；
6. 不要为无意义词汇创建链接；
7. 不要制造大量空页面；
8. 新概念、新人物、新事件、新观点稳定且会复用时，才创建新页面；
9. 每个 wiki 页面底部必须包含“相关页面”部分。
10. 当链接目标使用文件路径时，正文和相关列表应使用 `[[页面路径|显示名称]]` 格式，避免直接把路径暴露给读者。
11. 相关列表推荐使用 `- [[页面路径|显示名称]]：说明这个页面为什么相关。`，让链接文本显示页面标题或语义名称，路径只保留在链接内部。
12. 正文中只要引用已经存在的其他页面，包括来源页、概念页、人物页、事件页、观点页、时间线页、推导链页和主题页，都必须使用 Obsidian 双链；不要只写日期、标题或页面名纯文本。若日期实际指向某篇来源或页面，应写成 `[[sources/path|2023-07-21《标题》]]` 这类带显示名的链接。

示例：

```markdown
[[concepts/llm-wiki|LLM Wiki]] 与 [[concepts/obsidian-wiki|Obsidian-Wiki]] 的区别在于：
LLM Wiki 更像一种模式，而 Obsidian-Wiki 更像面向 Agent 的工作流框架。

[[people/michael-burry|Michael Burry]] 的做空行为被作者用作市场风险判断的一个信号，但该信号仍需结合 [[concepts/liquidity|流动性]] 和 [[concepts/market-sentiment|市场情绪]] 判断。

- [[views/lantingji-nonferrous-view|蓝庭集对有色金属的看法]]：这是一个 View Page，记录“谁怎么看”。
- [[reasoning/nonferrous-turning-point|有色金属拐点推导]]：这是对应的 Reasoning Page，拆解“为什么这样推”。
```

## 7. 命名规则

页面命名应清晰、稳定、可复用。

### 7.1 通用命名

优先使用：

```text
中文概念名.md
英文术语.md
中文说明 + 英文术语.md
```

避免：

```text
新建文档.md
未命名.md
总结1.md
文章整理.md
临时.md
```

同一概念、人物、事件、观点不得重复建立多个页面。

---

### 7.2 Event Page 命名

事件页建议使用：

```text
YYYY-MM-DD-事件标题.md
```

示例：

```text
2026-05-14-风险节点.md
2026-05-15-特朗普访华.md
```

如果日期不确定，可以使用：

```text
事件标题.md
```

---

### 7.3 People Page 命名

人物页建议使用人物常用名。

示例：

```text
Andrej Karpathy.md
Michael Burry.md
特朗普.md
蓝庭集.md
```

如果同名人物可能混淆，可在标题中加身份说明。

---

### 7.4 View Page 命名

观点页建议使用：

```text
人物名对主题的看法.md
机构名对主题的判断.md
作者名关于主题的观点.md
```

示例：

```text
蓝庭集对有色金属的看法.md
Karpathy对LLM Wiki的观点.md
某博主对AI与财政税收模式的判断.md
```

## 8. ingest workflow

当用户要求整理一篇新资料时，必须按以下流程执行。

### 第一步：读取规则

先读取：

1. `AGENTS.md`
2. `schema.md`
3. `page-types.md`
4. `index.md`

### 第二步：保存原始资料

将原始资料保存到 `sources/`。

根据资料类型选择子目录：

```text
sources/articles/
sources/posts/
sources/papers/
sources/screenshots/
sources/transcripts/
sources/webpages/
```

### 第三步：判断页面类型

根据 `page-types.md` 判断资料应该进入哪类结构化页面：

```text
concept
people
event
view
timeline
reasoning
topic
```

### 第四步：优先更新已有页面

在新建页面前，先检查 `wiki/` 中是否已有相关页面。

优先更新已有页面，避免重复。

### 第五步：提炼结构化知识

根据页面类型写入对应结构。

不得只做普通摘要。  
不得把资料整篇搬运到 `wiki/`。  
不得把观点页写成百科人物页。  
不得把推导链写成松散观点集合。

### 第六步：建立双链

为核心概念、人物、事件、观点、时间线、推导链和主题建立 Obsidian 双链。

### 第七步：更新 index.md

如果新增了重要页面，必须更新 `index.md`。

### 第八步：更新 log.md

每次修改后必须记录：

```markdown
## YYYY-MM-DD

- 操作：
- 修改文件：
- 新增文件：
- 资料来源：
- 说明：
```

## 11. 质量规则

Agent 输出必须满足以下质量要求：

1. 不做无来源断言；
2. 不把观点写成事实；
3. 不把推测写成结论；
4. 不重复创建相同概念、人物、事件、观点页面；
5. 不把文章摘要当成知识沉淀；
6. 不把 timeline 和 reasoning 混用；
7. 不把 view 和 reasoning 混用；
8. 不把 topic 当成所有内容的默认归宿；
9. 不随意删除旧内容；
10. 修改前尽量理解已有上下文；
11. 重要变更应更新 index 和 log；
12. 不确定的信息必须标注“待验证”。

---

## 12. 事实、观点、推测的区分

整理资料时，必须区分以下三类信息：

### 事实

可以被来源验证的信息。

示例：

```markdown
事实：某政策在某日期发布。
```

### 观点

作者或资料提供者的判断。

示例：

```markdown
观点：作者认为当前市场风险主要来自流动性收缩。
```

### 推测

基于现有信息的推导，但尚未被验证。

示例：

```markdown
推测：如果美债收益率继续上行，可能压制高估值资产。
```

## 13. View 与 Reasoning 的区分

当内容来自某个作者、博主、机构或人物时，必须判断它是 View 还是 Reasoning。

### View Page

记录：

```text
谁怎么看？
他说了什么？
他的判断是什么？
他的立场是什么？
```

### Reasoning Page

记录：

```text
这个判断是如何一步步推导出来的？
前提是什么？
中间变量是什么？
传导机制是什么？
什么条件会让它失效？
```

判断规则：

```text
重点是“谁怎么看” → View Page
重点是“为什么这么推” → Reasoning Page
```

二者可以同时存在，并互相链接。

## 14. People Page 维护 workflow

People Page 用于维护“人物画像”，不是资料堆放页，也不是 Topic Page 的替代品。

People Page 的重点是：

```text
这个人是谁？
这个人为什么值得进入知识库？
这个人的价值观、投资理念、判断气质、认知模式是什么？
这个人的观点适合如何使用？
这个人的观点有哪些边界和盲区？
```

具体观点、推导、事件和概念不得直接堆进 People Page。应拆分到对应页面，再通过双链回流到 People Page。

---

### 14.1 People Page 的维护原则

维护 People Page 时必须遵守：

1. People Page 先写“人”，再索引“内容”；
2. 不把 People Page 写成 Topic Page；
3. 不把人物页写成文章列表；
4. 不把具体观点全文写进人物页；
5. 不把具体推导链全文写进人物页；
6. 人物画像必须来自多个具体页面的归纳，不得凭单篇资料定性；
7. 对尚未被多篇资料支持的人物判断，必须标注“暂定”或“待验证”。

正确分工：

```text
人物画像 → People Page
具体观点 → View Page
具体推导 → Reasoning Page
稳定概念 → Concept Page
具体事件 → Event Page
问题域聚合 → Topic Page
```

---

### 14.2 People Page 的整理方法

维护人物页时，采用以下方法：

```text
先轻度自上而下建骨架；
再主要自下而上积累证据；
最后周期性回到人物页做归纳。
```

不要一开始就强行定义人物的价值观、性格、投资理念。

更稳妥的方式是：

1. 先建立 People Page 空模板；
2. 每篇新资料先按内容性质拆到 View / Reasoning / Concept / Event / Timeline / Topic；
3. 在这些具体页面中添加“相关人物：[[人物名]]”；
4. 每整理 5-10 篇相关页面后，回到 People Page 进行一次归纳；
5. 根据新增证据修正人物画像，持续迭代。

---

### 14.3 新建 People Page workflow

当知识库中出现一个值得长期跟踪的人物、作者、投资人、研究者、博主或观点来源时，按以下流程新建 People Page。

#### 第一步：确认是否需要新建

新建前必须检查：

1. `wiki/people/` 中是否已有同名或同义人物页；
2. 该人物是否会被后续复用；
3. 该人物是否是观点、推导、事件或概念的重要来源；
4. 是否可以只在 View Page 中作为观点来源，不必单独建 People Page。

只有当该人物具有长期复用价值时，才新建 People Page。

#### 第二步：建立空骨架

新建 People Page 时，先建立轻量骨架。

初始阶段允许大量章节标注：

```text
待归纳
待验证
暂无足够材料
```

不得在证据不足时过早写成确定人物画像。

#### 第三步：补充基础信息

优先补充：

- 身份；
- 平台或机构；
- 相关领域；
- 与本知识库的关系；
- 主要作为哪类知识来源。

基础信息必须尽量可追踪来源。

#### 第四步：建立初始双链

至少补充：

- 相关观点；
- 相关推导；
- 相关概念；
- 相关主题；
- 来源。

如果尚无对应页面，可以先保留待补充项，不要制造大量空链接。

---

### 14.4 新资料进入知识库时的人物回流 workflow

当整理一篇与某个人物相关的新资料时，不要直接把内容追加到 People Page。

应先判断资料本身属于哪种页面类型：

```text
具体观点 → View Page
因果推导 → Reasoning Page
稳定术语或模型 → Concept Page
单个事件 → Event Page
事件顺序 → Timeline Page
问题域组织 → Topic Page
```

然后在对应页面中添加人物关系。

推荐写法：

```markdown
## 相关人物

- [[people/人物名|人物名]]：说明该人物与本页面的关系。
```

最后再回到 People Page，在对应索引区补充该页面链接。

---

### 14.5 People Page 周期性归纳 workflow

每累计整理 5-10 篇与同一人物相关的页面后，必须回到 People Page 做一次归纳。

归纳时重点更新以下章节：

1. 核心价值观；
2. 投资理念 / 判断理念；
3. 人物性格与判断气质；
4. 认知模式；
5. 能力结构 / 技能树；
6. 风险偏好与仓位倾向；
7. 代表性判断模式；
8. 适用场景；
9. 使用边界与可能盲区；
10. 待补充问题。

归纳时必须回答：

```text
这个人反复强调什么？
这个人反复警惕什么？
这个人如何从信息推出结论？
这个人的判断偏进攻还是防守？
这个人的观点在哪些场景下有效？
这个人的观点在哪些情况下可能失效？
```

### 14.6 People Page 推荐结构

People Page 的可复制模板见：

- `templates/people-page-template.md`

维护人物页时，以该模板为结构基准；具体观点、推导、事件和概念仍应拆分到对应页面，再回链到 People Page。

---

### 14.7 价值观与投资理念的区分

维护 People Page 时，必须区分“价值观”和“投资理念”。

```text
价值观 = 这个人相信什么、重视什么、排斥什么。
投资理念 = 这个人如何把这些信念落实到投资判断里。
```

价值观关注底层取舍，例如：

- 优先保护什么；
- 愿意牺牲什么；
- 如何看待风险、收益、时间、确定性和不确定性；
- 反对什么样的市场行为或认知方式。

投资理念关注判断方法，例如：

- 如何判断一个资产是否值得买；
- 如何处理基本面、流动性、情绪、估值、周期、风险；
- 如何判断买入、持有、加仓、减仓、回避；
- 如何区分长期价值、阶段机会、情绪炒作和风险窗口。

不得把具体交易规则误写成价值观。

---

### 14.8 People Page 证据要求

人物画像中的判断必须有证据来源。

规则：

1. 单篇资料只能形成“观察”，不能直接形成人物定性；
2. 至少有 3 个相关页面支持时，才可以写成相对稳定的人物特征；
3. 如果只有 1-2 个页面支持，必须标注“暂定”；
4. 如果后续材料出现反例，必须修正人物画像；
5. 不得把个人印象写成确定结论。

推荐写法：

```markdown
### 风险敏感（暂定）

目前主要依据以下页面观察得到：

- [[views/...|观点页面A]]
- [[reasoning/...|推导页面B]]

该特征仍需更多材料验证。
```

---

### 14.9 People Page 禁止写法

不得这样维护 People Page：

1. 把人物页写成该人物所有文章的目录；
2. 把人物页写成 Topic Page；
3. 把具体观点全文写入人物页；
4. 把完整推导链写入人物页；
5. 用单篇文章给人物贴长期标签；
6. 把作者语气当作确定结论；
7. 忽略观点的时间、背景和适用条件；
8. 不链接 View / Reasoning / Concept / Event 页面；
9. 不标注不确定性；
10. 不记录来源。

---

### 14.10 People Page 更新触发条件

出现以下情况时，应检查是否需要更新 People Page：

1. 新增该人物的重要 View Page；
2. 新增该人物的重要 Reasoning Page；
3. 新增能体现人物判断模式的 Concept Page；
4. 新增该人物参与或评论的重要 Event Page；
5. 同一人物相关页面累计增加 5-10 篇；
6. 新材料推翻或修正了原有人物画像；
7. 用户要求整理或重构该人物；
8. Skill 需要更稳定地调用该人物作为判断来源。

---

### 14.11 People Page 与 index / log 的关系

新建重要 People Page 时，应更新 `index.md` 的人物入口。

对 People Page 做重要归纳、重构或人物画像修正时，必须更新 `log.md`。

日志应说明：

```markdown
## YYYY-MM-DD

### 操作类型

refactor / link / audit

### 修改文件

- `wiki/people/人物名.md`
- 相关 View / Reasoning / Concept / Event 页面

### 操作说明

说明本次更新了哪些人物画像、补充了哪些链接、修正了哪些判断。

### 后续待办

- 待验证的人物特征
- 待补充的观点页
- 待拆分的推导页
```

## 15. 推导链写作要求

当内容涉及因果关系时，必须写清楚推导链。

错误写法：

```markdown
市场可能下跌，因为流动性不好。
```

正确写法：

```markdown
1. 美债收益率上行；
2. 无风险收益率提高；
3. 高估值资产折现率上升；
4. 资金从风险资产流向债券；
5. 股市估值承压；
6. 因此市场可能出现调整。
```

---

## 16. 不确定性标注

以下情况必须标注不确定性：

1. 数据来源不明；
2. 作者观点较强；
3. 事件尚未发生；
4. 推导依赖多个假设；
5. 市场、政策、医疗、法律、金融等高变动领域；
6. 用户提供材料本身没有给出证据。

推荐写法：

```markdown
## 不确定性

- 该判断依赖于“流动性继续收缩”这一前提。
- 当前资料未提供完整数据来源。
- 该结论属于作者推测，尚需后续事件验证。
```

## 17. 禁止行为

Agent 不得执行以下行为：

1. 不得删除 `sources/` 中原始资料；
2. 不得覆盖用户已有重要页面；
3. 不得未经检查就新建重复页面；
4. 不得把未经验证的信息写成确定事实；
5. 不得把所有资料都整理成普通摘要；
6. 不得忽略 `schema.md`；
7. 不得忽略 `page-types.md`；
8. 不得修改与任务无关的大量文件；
9. 不得在没有必要时重构整个知识库；
10. 不得把临时内容写入长期页面；
11. 不得省略 `index.md` 和 `log.md` 更新。

---

## 18. 修改前检查

执行修改前，Agent 应先判断：

1. 是否已有相关页面；
2. 是否需要新建页面；
3. 新页面属于哪种页面类型；
4. 是否需要更新 index；
5. 是否需要记录 log；
6. 是否会影响已有双链；
7. 是否存在命名冲突；
8. 是否需要先向用户说明修改计划。

对于大规模修改，必须先输出修改计划，不得直接执行。

## 19. 推荐用户指令格式

```text
请按 Obsidian-Wiki ingest workflow 整理下面资料。
```

### 整理观点

```text
请将下面内容整理为 view 页面，重点记录：
观点来源、核心观点、观点依据、相关人物、相关事件、相关概念、不确定性。
```

### 整理推导类文章

```text
请将下面文章整理为 reasoning 页面，重点提取：
结论、前提、变量、推导链、风险点、不确定性。
```

### 整理人物

```text
请将这个人物整理为 people 页面，重点记录：
基本信息、人物背景、人物定位、核心价值观、投资理念 / 判断理念、人物性格与判断气质、认知模式、能力结构 / 技能树、风险偏好与仓位倾向、代表性判断模式、适用场景、使用边界与可能盲区。

不要把 People Page 写成 Topic Page；具体观点写入 View Page，具体推导写入 Reasoning Page，稳定概念写入 Concept Page，再回链到 People Page。
```

### 维护人物页

```text
请按 People Page 维护 workflow 更新这个人物页：
先检查相关 View / Reasoning / Concept / Event 页面，再归纳人物的价值观、投资理念、判断气质、认知模式、能力结构和使用边界。

如果证据不足，请标注“暂定”或“待验证”，不要过早定性。
```

### 整理事件

```text
请将这个事件整理为 event 页面，重点记录事件概述、时间、参与方、背景、影响、相关推导和所属时间线。
```

### 审计知识库

```text
请审计 llmwiki，检查重复页面、孤立页面、缺少来源页面和双链问题。
```

### 重构页面

```text
请按 schema.md 和 page-types.md 重构这个页面，保持原意，不改变 sources。
```

### 补充索引

```text
请检查 index.md 是否覆盖当前 wiki 页面，并补充缺失入口。
```

## 20. 最终目标

本知识库的最终目标是：

```text
把零散资料沉淀为可复用、可追踪、可推理、可持续演进的个人知识系统。
```

Agent 的价值不在于生成更多文字，而在于：

1. 降低知识整理成本；
2. 提高知识复用率；
3. 保持页面结构一致；
4. 建立知识之间的关系；
5. 帮助用户形成长期可维护的 Wiki。
