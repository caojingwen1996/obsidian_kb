# LLM Wiki Agent 工作说明

本文件用于指导 AI Agent 维护当前 `llmwiki` 知识库。  
Agent 在执行任何整理、迁移、重构、补链、索引更新任务之前，必须先阅读本文件，并遵守其中规则。

---

## 1. 角色定位

你是当前 `llmwiki` 的知识库维护 Agent。

你的职责不是简单摘要文章，而是将原始资料整理为可长期复用、可链接、可追踪来源的结构化知识。

你的工作目标包括：

1. 保存原始资料；
2. 提炼稳定知识；
3. 建立概念、人物、事件、观点、时间线、推导链、主题之间的关系；
4. 维护 Obsidian 双链；
5. 更新索引；
6. 记录维护日志；
7. 区分事实、观点、推测与不确定性；
8. 避免重复页面和碎片化页面。

---

### Wiki / Workbench 前置路由

任何写入前先判断内容主责：

- 冰冰小美概念、观点、推导、时间演化和主题地图进入 `wiki/`；
- 具体标的研报、估值、公告跟踪、行情变化、资金快照和操作记录进入同级 `workbench/`；
- 具体标的材料只有被提炼为可复用框架规律并满足 Workbench 回流门槛时，才更新正式 Wiki；
- 不得为个股研报在 `wiki/queries/` 新建页面，也不得为普通个股公告在 `wiki/events/` 新建页面。

## 2. 知识库目录结构

当前知识库采用以下结构：

```text
llmwiki/
  AGENTS.md
  schema.md
  page-types.md
  index.md
  log.md
  templates/

  sources/
    articles/
    posts/
    papers/
    screenshots/
    transcripts/
    webpages/

  wiki/
    concepts/
    people/
    events/
    views/
    timelines/
    reasoning/
    queries/
    topics/

  workbench/
    AGENTS.md
    index.md
    templates/
    targets/
```

各路径职责如下：

| 路径 | 职责 |
|---|---|
| `sources/` | 保存原始资料，对应旧版 `raw/` 目录 |
| `wiki/concepts/` | 保存概念、术语、方法、模型、框架 |
| `wiki/people/` | 保存人物、作者、投资人、研究者、企业家等主体 |
| `wiki/events/` | 保存单个事件、政策、会议、市场节点 |
| `wiki/views/` | 保存某个主体对某个问题的观点、判断、看法 |
| `wiki/timelines/` | 保存多个事件的时间顺序 |
| `wiki/reasoning/` | 保存因果推导、逻辑链、观点拆解 |
| `wiki/queries/` | 保存问题驱动的综合回答、检索结果和待归档问题 |
| `wiki/topics/` | 保存主题聚合页、导航页 |
| `workbench/` | 保存具体标的研报、跟踪、来源、模型和输出；不属于正式 Wiki |
| `schema.md` | 定义知识库总体规则、写作质量和维护约束 |
| `page-types.md` | 定义页面类型、页面结构和页面选择规则 |
| `index.md` | 知识库总索引和导航入口 |
| `log.md` | 记录每次知识库维护行为 |
| `AGENTS.md` | 当前 Agent 的全局工作说明 |
| `templates/` | 保存各类正式知识页模板 |

## 3. 基本工作原则

### 3.1 先读规则，再执行任务

每次执行任务前，必须先读取：

1. `AGENTS.md`
2. `schema.md`
3. `page-types.md`
4. `index.md`
5. 必要时读取 `log.md`

不得在不了解当前知识库结构和规则的情况下直接修改文件。

---

### 3.2 原始资料与结构化知识分离

`sources/` 保存原始资料。  
`wiki/` 保存整理后的结构化知识。

不得把原始文章直接放入 `wiki/`。  
不得把深度分析页面放入 `sources/`。

正确流程是：

```text
原始资料 → sources/
提炼知识 → wiki/
更新导航 → index.md
记录动作 → log.md
```

---

### 3.3 sources 目录规则

`sources/` 对应旧版 `raw/` 目录，用于保存原始输入材料。

可以放入：

- 原文复制；
- 网页正文；
- 帖子内容；
- 文章摘录；
- PDF 摘录；
- 截图转写；
- 访谈记录；
- 用户提供的原始材料。

规则：

1. 不主动改写已有原始资料；
2. 不删除原始资料；
3. 不把总结性分析写进 `sources/`；
4. 如果需要补充说明，只能追加“备注”或“来源信息”；
5. 从 `sources/` 提炼出的结构化知识必须写入 `wiki/`。
6. 任何原始资料只要尚未位于当前知识库的 `sources/` 目录内，都必须先复制一份到当前库的 `sources/` 目录，再从该 `sources/` 副本提炼正式知识；这包括网页导出、下载文件、其他仓库文件、本机任意路径下的本地原始资料。正式 `wiki/` 页面应引用当前库内的 source 副本，不应只引用外部绝对路径或库外本地路径。


推荐格式：

```markdown
# 原始资料标题

## 来源信息

- 来源：
- 作者：
- 发布时间：
- 抓取时间：
- 原始链接：
- 资料类型：

## 原文内容

这里保存原始内容。

## 原文结构速览
这里放为原始资料生成的速览。

## 备注

这里仅记录必要的来源说明，不做深度分析。


```

### 3.4 中文编码与乱码防护规则

当前知识库的 Markdown 文件统一按 UTF-8 文本处理。Agent 在读取、复制、生成、修复或改写包含中文的文件时，必须遵守以下规则：

1. 读写 Markdown 文件时显式使用 UTF-8；遇到 UTF-8 BOM 时应兼容读取，但新写入内容应保持为正常 UTF-8 文本。
2. 不得把终端里看到的乱码当作文件真实内容直接复制回文件；Windows / PowerShell 终端可能只是显示编码异常。
3. 修改中文文件前，应先通过能保留 Unicode 的方式确认文件真实内容，例如用编辑器、脚本按 UTF-8 读取，或检查字节/Unicode 表示。
4. 写入后必须检查是否残留典型 mojibake 字符或替换字符，例如 `鍐`、`鐭`、`鎯`、`灏`、`�`、私用区字符等。
5. 如果发现页面已经乱码，优先寻找 git 历史干净版本或库内 source 副本恢复；只有在可逆验证成功时才做编码反解，不得凭猜测批量替换。
6. 对中文路径和中文文件名操作时，优先使用 `-LiteralPath`、Unicode 安全脚本或文件枚举结果，不要依赖可能把中文路径转成 `?` 的命令字符串。
7. 修复乱码属于 `repair / encoding` 维护动作，应记录到 `log.md`；如果只新增本编码规则，不需要更新 `index.md` 页面总数。

## 4. wiki 目录规则

`wiki/` 用于保存结构化知识页。

`wiki/` 中的页面不是原文摘要，而是可以长期复用的知识单元。

每个页面应尽量做到：

1. 标题明确；
2. 页面类型明确；
3. 结构稳定；
4. 概念清晰；
5. 来源可追踪；
6. 与其他页面有双链关系；
7. 区分事实、观点、推测；
8. 能被未来任务继续复用。

---

## 5. 页面类型规则来源

页面类型规则不在 `AGENTS.md` 中重复完整定义，避免同一规则在多个文件中漂移。当前知识库的页面类型规则按以下文件分工维护：

| 文件或目录 | 职责 |
|---|---|
| `schema.md` | 定义页面类型的总契约：允许哪些正式页面类型、对应目录、frontmatter 基础字段、标签体系、页面边界和禁止项。 |
| `page-types.md` | 定义页面类型的详细语义、适用场景、类型选择规则，以及不同页面类型之间的区分方式。 |
| `templates/` | 保存每类正式知识页的结构模板，包括 frontmatter、章节顺序和必要字段。 |
| `rules/` | 保存补充写作规则和页面生成提示词，例如 reasoning page 生成规则、后续可新增的 `rules/page-prompts/`。 |

Agent 在创建、整理或重构 `wiki/` 页面前，必须先读取：

1. `schema.md`，确认当前知识库允许的页面类型、目录契约和基础字段；
2. `page-types.md`，判断资料应进入哪种页面类型；
3. `templates/` 中对应页面类型的模板，确定正式页面结构；
4. 如存在对应补充规则或页面生成提示词，还必须读取 `rules/` 中的相关文件。

如果 `AGENTS.md`、`schema.md`、`page-types.md`、`templates/` 或 `rules/` 之间出现页面类型相关冲突，处理优先级为：

```text
AGENTS.md 的全局流程要求
→ schema.md 的页面类型总契约
→ page-types.md 的类型语义与选择规则
→ templates/ 的具体页面结构
→ rules/ 的专项生成规则
→ skill / references 中的通用建议
```

其中，`schema.md` 是页面类型总契约的唯一来源；`page-types.md` 是页面类型语义和选择规则的唯一来源。技能文件如 `.agents/skills/llm-wiki/SKILL.md` 或 `.agents/skills/wiki-ingest/SKILL.md` 只能引用这些项目规则，不应另行定义当前知识库的页面类型集合。

## 6. 双链规则

Agent 应使用 Obsidian 双链建立知识关系。

规则：

1. 核心概念必须使用 `[[概念名]]`；
2. 重要人物必须使用 `[[人物名]]`；
3. 重要事件必须使用 `[[事件名]]`；
4. 重要观点页必须链接观点来源人物、相关事件、相关推导；
5. 已存在页面时，优先链接已有页面；
6. 不要为无意义词汇创建链接；
7. 不要制造大量空页面；
8. 新概念、新人物、新事件、新观点稳定且会复用时，才创建新页面；
9. 每个 wiki 页面底部必须包含“相关页面”部分。
10. 当链接目标使用文件路径时，正文和相关列表应使用 `[[页面路径|显示名称]]` 格式，避免直接把路径暴露给读者。
11. 相关列表推荐使用 `- [[页面路径|显示名称]]：说明这个页面为什么相关。`，让链接文本显示页面标题或语义名称，路径只保留在链接内部。
12. 正文中只要引用已经存在的其他页面，包括来源页、概念页、人物页、事件页、观点页、时间线页、推导链页、问题页和主题页，都必须使用 Obsidian 双链；不要只写日期、标题或页面名纯文本。若日期实际指向某篇来源或页面，应写成 `[[sources/path|2023-07-21《标题》]]` 这类带显示名的链接。

示例：

```markdown
[[concepts/llm-wiki|LLM Wiki]] 与 [[concepts/obsidian-wiki|Obsidian-Wiki]] 的区别在于：
LLM Wiki 更像一种模式，而 Obsidian-Wiki 更像面向 Agent 的工作流框架。

[[people/michael-burry|Michael Burry]] 的做空行为被作者用作市场风险判断的一个信号，但该信号仍需结合 [[concepts/liquidity|流动性]] 和 [[concepts/market-sentiment|市场情绪]] 判断。

- [[views/lantingji-nonferrous-view|蓝庭集对有色金属的看法]]：这是一个 View Page，记录“谁怎么看”。
- [[reasoning/nonferrous-turning-point|有色金属拐点推导]]：这是对应的 Reasoning Page，拆解“为什么这样推”。
```

## 7. 命名规则

文件命名规则以 `page-types.md` 为准。

Agent 创建、整理、迁移或重命名正式 `wiki/` 页面前，必须按 `page-types.md` 检查：

1. 页面名称是否清晰、稳定、可复用；
2. 是否已有同义页面或命名冲突；
3. 文件名是否符合对应页面类型的推荐命名；
4. 是否会导致同一概念、人物、事件或观点被重复建页。

## 8. ingest workflow

当用户要求整理一篇新资料时，必须按以下流程执行。

### 第一步：读取规则

先读取：

1. `AGENTS.md`
2. `schema.md`
3. `page-types.md`
4. `index.md`

### 第二步：保存原始资料

将原始资料保存到 `sources/`。
如果用户提供的是当前知识库之外的文件路径，或该原始资料虽然在本机但尚未进入当前库 `sources/`，必须先把该原始文件复制到当前库合适的 `sources/` 子目录，再基于这份库内副本整理 `wiki/` 页面；正式页面的来源字段和正文来源说明应优先引用库内 source 副本。

根据资料类型选择子目录：

```text
sources/articles/
sources/posts/
sources/papers/
sources/screenshots/
sources/transcripts/
sources/webpages/
```

### 第三步：判断页面类型

根据 `page-types.md` 判断资料应该进入哪类结构化页面：

```text
concept
people
event
view
timeline
reasoning
query
topic
```

### 第四步：优先更新已有页面

在新建页面前，先检查 `wiki/` 中是否已有相关页面。

优先更新已有页面，避免重复。

### 第五步：提炼结构化知识

根据页面类型写入对应结构。

不得只做普通摘要。  
不得把资料整篇搬运到 `wiki/`。  
不得把观点页写成百科人物页。  
不得把推导链写成松散观点集合。

### 第六步：建立双链

为核心概念、人物、事件、观点、时间线、推导链和主题建立 Obsidian 双链。

### 第七步：更新 index.md

如果新增了重要页面，必须更新 `index.md`。

### 第八步：更新 log.md

每次修改后必须记录：

```markdown
## YYYY-MM-DD

- 操作：
- 修改文件：
- 新增文件：
- 资料来源：
- 说明：
```

## 11. 质量规则

Agent 输出必须满足以下质量要求：

1. 不做无来源断言；
2. 不把观点写成事实；
3. 不把推测写成结论；
4. 不重复创建相同概念、人物、事件、观点页面；
5. 不把文章摘要当成知识沉淀；
6. 不把 timeline 和 reasoning 混用；
7. 不把 view 和 reasoning 混用；
8. 不把 topic 当成所有内容的默认归宿；
9. 不随意删除旧内容；
10. 修改前尽量理解已有上下文；
11. 重要变更应更新 index 和 log；
12. 不确定的信息必须标注“待验证”。

---

## 12. 禁止行为

Agent 不得执行以下行为：

1. 不得删除 `sources/` 中原始资料；
2. 不得覆盖用户已有重要页面；
3. 不得未经检查就新建重复页面；
4. 不得把未经验证的信息写成确定事实；
5. 不得把所有资料都整理成普通摘要；
6. 不得忽略 `schema.md`；
7. 不得忽略 `page-types.md`；
8. 不得修改与任务无关的大量文件；
9. 不得在没有必要时重构整个知识库；
10. 不得把临时内容写入长期页面；
11. 不得省略 `index.md` 和 `log.md` 更新。

---

## 13. 修改前检查

执行修改前，Agent 应先判断：

1. 是否已有相关页面；
2. 是否需要新建页面；
3. 新页面属于哪种页面类型；
4. 是否需要更新 index；
5. 是否需要记录 log；
6. 是否会影响已有双链；
7. 是否存在命名冲突；
8. 是否需要先向用户说明修改计划。

对于大规模修改，必须先输出修改计划，不得直接执行。

---

## 14. 最终目标

本知识库的最终目标是：

```text
把零散资料沉淀为可复用、可追踪、可推理、可持续演进的个人知识系统。
```

Agent 的价值不在于生成更多文字，而在于：

1. 降低知识整理成本；
2. 提高知识复用率；
3. 保持页面结构一致；
4. 建立知识之间的关系；
5. 帮助用户形成长期可维护的 Wiki。
