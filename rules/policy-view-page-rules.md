## Policy View 页面生成规则

本文件只定义政策解读型 View Page 的观点提炼要求。frontmatter、页面命名、目录位置、来源字段、相关页面、双链格式和更新流程，以 `schema.md`、`page-types.md`、`AGENTS.md` 和 `templates/view-page-template.md` 为准。

当整理 policy view 页面时，不能直接总结政策原文，也不能把政策传导链完整塞进观点页。必须把来源中的“作者如何理解政策”提炼成可复用的阶段性观点，并明确区分政策事实、作者观点、整理者推断和待验证信息。

### 1. 识别页面定位

先判断该内容是否真的适合进入政策解读型 View Page。

适合进入政策解读型 View Page 的内容通常满足以下条件：

- 有明确观点来源，例如某位作者、机构、博主或研究者；
- 重点是“这个主体如何理解某项政策”；
- 政策对象可以是政策、会议、制度、改革、财政安排、货币工具或监管变化；
- 作者不仅复述政策内容，还给出政策目的、作用路径、约束条件或市场影响判断；
- 该观点可以服务后续 Concept Page、Reasoning Page、Topic Page 或时间线复盘。

不适合进入政策解读型 View Page 的内容应转入其他页面类型：

- 只解释政策术语是什么时，进入 Concept Page；
- 只记录政策本身发生了什么时，进入 Event Page；
- 重点是完整因果传导链时，进入 Reasoning Page；
- 多篇政策材料的主题地图，进入 Topic Page；
- 按时间记录政策节点，进入 Timeline Page；
- 保存政策原文、公告、会议纪要或文章原文，进入 Source。

### 2. 明确政策对象

必须先说清楚作者解读的政策对象是什么。

政策对象应包含：

- 政策、会议、工具或制度名称；
- 相关主体，例如央行、财政、国务院、部委、地方政府、交易所、监管机构；
- 时间范围或发布节点；
- 政策作用领域，例如财政、货币、信用、流动性、产业、监管、资本市场、房地产、外贸、科技等。

避免：

- 只写“政策利好”而不说明政策对象；
- 把多个不同政策混成一个模糊对象；
- 用市场结果倒推政策含义；
- 把尚未落地的传闻写成确定政策。

### 3. 提炼作者核心判断

政策解读型 View Page 的核心是“谁怎么看”。

必须提炼作者对政策的判断，包括：

- 作者认为政策要解决什么问题；
- 作者认为政策采用了什么工具或路径；
- 作者认为政策改变了哪些变量；
- 作者认为政策会影响哪些主体、产业、资产或市场结构；
- 作者认为政策的约束条件和不确定性是什么；
- 作者是否把该政策放入更大的国情、周期、产业或风险框架中。

推荐表述：

```text
作者认为……
作者将该政策理解为……
作者把这项政策放入……框架中解释。
```

避免：

- 把作者判断写成政策事实；
- 把整理者推断写成作者原文观点；
- 把政策方向直接写成确定行情结论；
- 把政策规模直接等同于二级市场资金流入。

### 4. 区分政策事实、作者观点和整理者推断

必须在正文中区分三类信息：

- 政策事实：来源可验证的政策、会议、制度、工具、时间和主体；
- 作者观点：作者对政策目的、传导路径、影响范围和市场含义的判断；
- 整理者推断：为连接既有知识库页面而做出的归纳、对照或边界补充。

整理者推断应明确标注为“整理者归纳”或“待验证”，不能伪装成作者原文。

如果原文没有给出数据或政策原文链接，应写明“待核验”。

### 5. 拆解政策要解决的问题

政策解读页必须回答：作者认为该政策针对什么问题。

常见问题包括：

- 债务压力；
- 通缩压力；
- 信用收缩；
- 流动性不足；
- 财政空间受限；
- 房地产旧周期出清；
- 产业升级或产业安全；
- 外部竞争或地缘风险；
- 资本市场融资、定价或信心问题；
- 居民、企业、地方政府或金融机构行为失衡。

不要只写“刺激经济”“利好市场”这类过宽表述。

### 6. 提炼解读框架

必须说明作者从哪些维度理解该政策。

常见维度包括：

- 财政；
- 货币；
- 信用；
- 流动性；
- 债务；
- 汇率；
- 利率；
- 产业；
- 资产价格；
- 风险偏好；
- 国际竞争；
- 制度改革；
- 资本市场定位。

解读框架应服务作者观点，不要机械把所有维度都列一遍。

### 7. 保留简版传导路径

政策解读页可以保留简版传导路径，但不承担完整 Reasoning Page 职责。

推荐格式：

```text
政策对象
-> 改变某项财政、货币、信用、产业或监管条件
-> 改变资金数量、价格、流向、风险偏好或资源配置
-> 改变机构行为、产业预期或市场信心
-> 形成作者判断中的市场含义
```

如果传导链超过 5 步，涉及多层变量、多个中介环节或需要验证信号和失效条件，应另建或链接 Reasoning Page。

### 8. 提炼关键变量

必须列出后续需要观察的变量。

政策类观点常见观察变量包括：

- 政策落地时间；
- 政策规模；
- 财政资金来源；
- 央行工具变化；
- 信用扩张或收缩；
- 利率和汇率变化；
- 资金流向；
- 产业订单、产能、利润或价格；
- 资本市场成交、融资、ETF、外资、险资、公募、私募等资金主体行为；
- 政策是否被市场提前定价；
- 政策是否被实体承接。

观察变量必须与该政策对象和作者判断有关，不要泛泛列宏观指标。

### 9. 标注约束条件

必须记录政策传导可能受到的限制。

常见约束包括：

- 政策规模不足；
- 资金只在金融体系内空转；
- 地方执行滞后；
- 企业利润没有兑现；
- 居民或企业信心不足；
- 债务压力没有缓解；
- 外部风险打断传导；
- 市场已经提前定价；
- 资金流向与政策目标不一致；
- 作者判断依赖的前提被新数据证伪。

约束条件是政策解读页的核心部分，不能只写正面影响。

### 10. 说明市场含义

政策解读页可以记录作者如何把政策连接到市场，但不得写成直接买卖建议。

应说明：

- 作者认为政策可能影响哪些资产、行业、风格或市场结构；
- 这种影响是通过流动性、信用、风险偏好、产业承接还是制度变化发生；
- 当前判断属于作者观点、阶段性判断还是整理者归纳；
- 需要哪些后续信号验证。

禁止：

- 看到政策就直接给出买入、卖出、加仓、减仓结论；
- 把政策口号直接等同于产业利润兑现；
- 把政策工具直接等同于行情启动；
- 把单一政策解释成全部市场上涨原因；
- 忽略价格是否已经反应。

### 11. 说明与其他页面的拆分关系

政策解读型 View Page 应和其他页面保持边界。

- 与 Concept Page：Concept Page 解释政策概念是什么，Policy View 记录作者如何理解该政策。
- 与 Reasoning Page：Reasoning Page 拆完整传导链，Policy View 只保留观点摘要和简版传导路径。
- 与 Event Page：Event Page 记录政策事件本身，Policy View 记录作者对该事件的看法。
- 与 Timeline Page：Timeline Page 记录多个政策节点的先后顺序，Policy View 记录某一主体在某一阶段的判断。
- 与 Topic Page：Topic Page 聚合一个政策主题下的多篇观点、推导、事件和概念，Policy View 不承担地图职责。
- 与 Source：Source 保存原始政策、原文或摘录，Policy View 只引用来源，不全文复制来源。

### 12. 写作质量检查

完成政策解读型 View Page 前，必须检查：

- 是否明确页面是 `type: view`；
- 是否在需要时使用 `view_type: policy_interpretation`；
- 是否说明观点来源人物或机构；
- 是否说明政策对象；
- 是否说明作者核心判断；
- 是否说明政策要解决的问题；
- 是否说明作者的解读框架；
- 是否保留简版传导路径，并把复杂传导拆给 Reasoning Page；
- 是否列出关键观察变量；
- 是否列出约束条件和不确定性；
- 是否区分政策事实、作者观点和整理者推断；
- 是否避免写成投资建议；
- 是否链接相关人物、概念、推导、主题、事件和来源；
- 是否与已有 View Page、Reasoning Page、Concept Page 或 Topic Page 重复。
