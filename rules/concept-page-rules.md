## Concept 页面生成规则

本文件只定义 Concept Page 的概念提炼要求。frontmatter、页面命名、目录位置、来源字段、相关页面、双链格式和更新流程，以 `schema.md`、`page-types.md`、`AGENTS.md` 和 `templates/concept-page-template.md` 为准。

当整理 concept 页面时，不能直接总结原文，也不能把单篇文章观点包装成概念。必须把来源中的稳定概念、方法、模型、框架或判断工具提炼成可复用的知识单元。

### 1. 识别概念定位

先判断该内容是否真的适合进入 Concept Page。

适合进入 Concept Page 的内容通常满足以下条件：

- 它解释一个稳定概念、方法、模型、框架或判断工具；
- 它可以被多个 View Page、Reasoning Page、Topic Page 或后续资料复用；
- 它不只依赖单一事件、单一观点或一次性语境；
- 它能够回答“这是什么”“解决什么问题”“如何使用”“边界在哪里”。

不适合进入 Concept Page 的内容应转入其他页面类型：

- 重点是“谁怎么看”时，进入 View Page；
- 重点是“为什么这样推导”时，进入 Reasoning Page；
- 重点是“发生了什么”时，进入 Event Page；
- 重点是“先后顺序”时，进入 Timeline Page；
- 重点是聚合一个问题域时，进入 Topic Page。

### 2. 提炼概念定义

必须用简洁、准确、可复用的语言定义概念。

定义应包含：

- 概念指向的对象或现象；
- 概念试图解决的问题；
- 概念在相关体系中的作用；
- 概念的适用范围。

推荐格式：

> 概念定义：所谓「……」，指的是在……条件下，用来判断 / 解释 / 组织……的一种概念或方法。

避免：

- 写成文章读后感；
- 写成百科式泛泛解释；
- 只复述作者原句，不做知识库语境下的工作定义；
- 把整理者推断写成来源明确提出的定义。

### 3. 压缩核心本质

必须用一句话压缩这个概念的本质。

推荐格式：

```text
概念本质 = 核心动作 + 判断对象 + 实践目标
```

示例结构：

```text
底线思维 = 先推演最坏情形 + 再判断自身承受力 + 最后决定是否进入或持有
```

这句话应能帮助后续页面快速引用本概念，而不是替代完整定义。

### 4. 拆解核心问题

必须整理这个概念在使用时需要回答的问题。

问题清单应围绕概念本身生成，而不是机械套模板。

常见问题包括：

- 要判断什么？
- 要排除什么？
- 要观察什么？
- 要比较什么？
- 要等待什么？
- 要验证什么？
- 要在什么条件下行动？
- 什么情况说明这个概念不适用？

### 5. 提炼关键要素

必须拆解概念内部的关键要素。

每个要素需要说明：

- 这个要素是什么意思；
- 它在概念中承担什么作用；
- 它和其他要素之间是什么关系；
- 它是否来自来源明确表述，还是整理者归纳。

推荐结构：

```markdown
### 要素一：……

- 含义：
- 作用：
- 来源依据：
- 与其他要素的关系：
```

不要把要素拆解写成完整因果推导。完整推导链应放入 Reasoning Page。

### 6. 转化为使用检查表

如果来源支持，应把概念转化为可执行的判断检查表。

推荐格式：

| 检查项 | 要判断的问题 | 通过标准 | 不通过时的含义 |
|---|---|---|---|
| 检查项 A |  |  |  |
| 检查项 B |  |  |  |
| 检查项 C |  |  |  |

检查表只在概念本身具有实践用途时生成。纯解释性概念不必强行生成行动检查表。

### 7. 说明体系位置

必须说明该概念在当前知识体系中的位置。

需要回答：

- 它是上游概念、中游判断工具，还是下游执行规则；
- 它依赖哪些概念；
- 它支撑哪些 View Page、Reasoning Page 或 Topic Page；
- 它是否属于某个人物、作者、体系或方法论的一部分；
- 它是否可以进一步生长出 Reasoning Page、View Page，或 `rule` 类型的 Concept Page。

注意：`rule` 是 Concept Page 文件名中的页面用途，不是当前知识库的独立正式页面类型。

### 8. 说明概念关系

必须说明该概念与其他核心概念的关系。

优先整理以下关系：

- 上位概念：这个概念属于哪个更大的框架；
- 下位概念：它可以拆出哪些更具体的概念或规则；
- 同级概念：它与哪些相邻概念共同解释同一问题；
- 对立概念：它反对或纠正了哪种常见理解；
- 支撑关系：它支撑哪些观点、推导或主题。

不要制造大量低价值关系。只有能帮助理解、检索或后续复用的关系才写入页面。

### 9. 明确概念边界

必须说明这个概念不是什么，以及它不能解决什么。

边界应至少覆盖：

- 它不等同于什么；
- 它不能解释什么；
- 它适用的前提是什么；
- 它失效的条件是什么；
- 它与相似概念的区别是什么；
- 它最容易被误用成什么。

边界不是附加段落，而是防止概念泛化和误用的核心部分。

### 10. 提炼使用场景

如果来源支持，应整理该概念适合在哪些场景使用。

推荐格式：

```markdown
### 场景一：……

- 适用条件：
- 观察重点：
- 输出判断：
- 不适用情形：
```

场景必须来自来源内容或可被来源合理支持。不要为了完整而虚构应用场景。

### 11. 提炼观察信号

如果该概念用于判断现实问题，应把抽象概念转化为可观察信号。

观察信号应按领域选择，不要机械套用。

投资、宏观或产业类概念可考虑：

- 时间信号；
- 价格信号；
- 政策信号；
- 情绪信号；
- 资金信号；
- 产业信号；
- 供需信号；
- 竞争格局信号；
- 叙事扩散信号。

非投资类概念应使用对应领域的观察信号，不要强行套用市场指标。

### 12. 标注来源依据与推断

必须区分来源明确提出的内容与整理者归纳的内容。

推荐结构：

```markdown
## 原文依据

- 来源明确提出：
- 来源间接支持：
- 整理者归纳：
- 待验证：
```

来源没有直接说明的概念定义、关系或使用方法，必须标注为整理者归纳或待验证。

### 13. 提炼原子化笔记

可以在页面末尾提炼 8-15 条可被其他页面引用的原子化笔记。

要求：

- 每条只表达一个判断；
- 语言短；
- 概念清楚；
- 能独立被引用；
- 不脱离来源依据；
- 不把推测写成事实。

### 14. 写作质量检查

完成 Concept Page 前，必须检查：

- 是否把概念写成了文章摘要；
- 是否把作者观点写成了稳定概念；
- 是否把完整推导链塞进了 Concept Page；
- 是否说明了概念来源和整理者归纳边界；
- 是否说明了概念解决的问题；
- 是否说明了概念边界和误用方式；
- 是否能被后续 View Page、Reasoning Page 或 Topic Page 复用；
- 是否与已有同义概念页重复。
