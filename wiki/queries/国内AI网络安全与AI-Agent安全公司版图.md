# 国内 AI 网络安全与 AI Agent 安全公司版图

## 1. 原始问题

> 国内做 AI 网络安全和 AI Agent 安全的公司有哪些？

## 2. 先区分两条赛道

“AI 网络安全”常混合两个相反方向：

1. **AI for Security**：用大模型或智能体做告警研判、威胁检测、漏洞分析和自动响应。
2. **Security for AI**：保护模型、RAG、AI 应用和 Agent，防提示词注入、越狱、数据泄露、工具越权、MCP/Skill 投毒与失控执行。

真正的 **Agent 安全** 不应只停在内容审核，还应覆盖身份、最小权限、工具调用、运行时隔离、人工确认和全链路审计。^[inferred]

## 4. 选型时怎么缩小名单

- **已有阿里云、腾讯云或百度云技术栈**：先测对应云厂商产品，集成成本通常较低。^[inferred]
- **政企、金融、能源等强调私有化和合规交付**：优先比较 360、奇安信、绿盟、安恒、亚信安全、深信服。
- **重点是 Agent 身份、权限和责任链**：重点比较亚信安全、象信 AI，并检查其与现有 IAM/PAM 的集成。
- **重点是 API、敏感数据外发和模型调用审计**：重点比较青笠科技及云厂商的 Agent 网关。
- **重点是代码执行或高危工具隔离**：重点测试 360、Tenbox 以及任何能提供进程/文件/网络级沙箱的方案。
- **重点是上线前红队评测**：重点比较长亭、瑞莱智慧，以及奇安信、绿盟、安恒的评估模块。

## 5. 必须做的 PoC 检查

1. 间接提示词注入能否跨网页、邮件、文档和 RAG 被识别。
2. MCP/Skill 被替换、描述投毒或版本漂移时能否发现。
3. Agent 是否有独立身份、短期凭证、最小权限和逐次授权。
4. 文件、网络、Shell、数据库写操作能否沙箱化或由 HITL 阻断。
5. 是否能重放“用户指令 → 模型决策 → 工具参数 → 执行结果”的完整链路。
6. 中文业务语料下的误报、漏报、延迟和吞吐是否可接受。
7. 私有化部署、国产化环境、日志留存和数据不出域是否满足要求。
