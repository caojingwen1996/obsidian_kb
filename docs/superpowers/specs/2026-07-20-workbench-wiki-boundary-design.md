# Workbench 与 Wiki 边界设计

## 1. 背景

当前仓库最初用于整理冰冰小美的知识框架，但 `wiki/queries/` 已混入 14 份机构级个股决策研报，相关个股来源、HTML 输出和资金面分析也分散在 `sources/`、`hot.md`、`index.md` 与部分正式 Wiki 页面中。

这导致两类生命周期不同的内容共用同一知识层：

- 冰冰小美框架需要长期稳定、可复用、可形成概念与推导关系；
- 个股研究需要持续更新价格、财报、估值、事件和操作结论，具有高时效性。

本设计在仓库根目录新增与 `wiki/` 同级的 `workbench/`，把个股研究与操作跟踪迁出正式知识层，同时保留从工作台向框架库选择性回流可复用知识的通道。

## 2. 目标

1. 恢复 `wiki/` 作为冰冰小美知识框架的长期知识层。
2. 将个股研报、跟踪节点、估值模型和操作记录集中到 `workbench/`。
3. 不把 `workbench` 工件伪装成 Query、Event 或 Timeline 等正式 Wiki 页面。
4. 保留已有研报、来源和反向链接的可追踪性，不丢失用户内容。
5. 建立明确的知识回流门槛，避免个股噪声再次污染框架库。

## 3. 非目标

- 本次不重写14份现有研报的研究结论。
- 本次不重新计算个股估值、DCF 或交易动作。
- 本次不改变冰冰小美 Concept、View、Reasoning、Timeline 和 Topic Page 的内容模型。
- 本次不创建第二个仓库或第二个 Obsidian Vault。
- 本次不迁移与个股研究无关的来源、自动化结果或正式知识页。

## 4. 根目录结构

```text
llmwiki/
├─ AGENTS.md
├─ schema.md
├─ page-types.md
├─ index.md
├─ log.md
├─ sources/                 # 冰冰小美及正式 Wiki 使用的原始来源
├─ wiki/                    # 冰冰小美正式知识层
├─ workbench/               # 个股研究与动态决策工作域
│  ├─ AGENTS.md             # 工作台边界、更新和回流规则
│  ├─ index.md              # 工作台导航，不并入根 index.md
│  ├─ templates/
│  │  ├─ target-index-template.md
│  │  ├─ equity-report-template.md
│  │  └─ tracking-node-template.md
│  └─ targets/
│     └─ 601766-中国中车/
│        ├─ index.md
│        ├─ reports/
│        │  └─ 2026-07-16-1334-中国中车机构级决策研报.md
│        ├─ tracking/
│        │  └─ 2026-07-20-增持公告与操作调整.md
│        ├─ sources/
│        ├─ models/
│        └─ outputs/
└─ docs/
```

`workbench/` 与 `wiki/` 必须保持同级。不得创建 `wiki/workbench/`，也不得继续把新个股研报写入 `wiki/queries/`。

## 5. 两个区域的职责

### 5.1 `wiki/`

`wiki/` 只承载可长期复用的冰冰小美知识：

- 专有概念和框架；
- 作者的阶段观点；
- 可复用推导链；
- 框架演化时间线；
- 主题地图和问题答案；
- 能说明框架的精选案例。

正式页面继续遵守 `schema.md` 和 `page-types.md` 定义的八种页面类型。

### 5.2 `workbench/`

`workbench/` 承载面向具体标的和当前决策的工件：

- 个股机构级研报；
- 财报、公告和价格变化后的跟踪节点；
- DCF、相对估值和敏感性模型；
- 行情、资金面和风险快照；
- 当前动作及其升级、降级、证伪条件；
- HTML、Excel、图表等研究输出。

这些工件不是正式 Wiki 页面，不使用 `type: query`、`type: event` 或 `type: timeline` 冒充知识页。

## 6. 标的目录契约

每个标的使用稳定的“证券代码-公司简称”目录：

```text
workbench/targets/<证券代码>-<公司简称>/
```

目录内文件职责如下：

| 路径 | 职责 |
|---|---|
| `index.md` | 标的仪表盘，链接当前研报、最近动作、跟踪节点和下一验证点 |
| `reports/` | 完整研究报告；同一研究问题保留一个当前权威版本并原地更新 |
| `tracking/` | 按日期保存改变判断或操作逻辑的增量节点 |
| `sources/` | 只服务该标的研究的公告、财报、行情和检索快照 |
| `models/` | DCF、相对估值、敏感性分析等模型文件 |
| `outputs/` | HTML、Excel、图表等可交付输出 |

共享来源如果已经支撑正式 Wiki 页面，则继续保留在根 `sources/`，不得为迁移而破坏其现有来源链。只服务单一标的的来源可以迁入对应标的的 `sources/`。

## 7. 跟踪节点规则

只有满足下列任一条件的信息才创建 `tracking/` 节点：

- 改变 buy / sell / reduce / wait / observe / review 动作；
- 强化、减弱或证伪原投资逻辑；
- 改变公允价值区间或关键估值参数；
- 改变资金、情绪、竞争格局或风险状态；
- 形成明确的下一验证节点。

普通新闻、重复公告和不改变判断的信息只进入来源或标的索引的简短记录，不单独生成跟踪页。

标准跟踪节点至少包含：

```text
触发事件
新增证据
原逻辑状态：强化 / 不变 / 减弱 / 证伪 / 证据不足
估值影响
操作影响
下一验证节点
来源
```

例如，中国中车增持公告应进入：

```text
workbench/targets/601766-中国中车/tracking/2026-07-20-增持公告与操作调整.md
```

公告原文和行情快照进入同一标的的 `sources/`；当前有效动作同时回写该标的 `reports/` 和 `index.md`。

## 8. Wiki 回流规则

工作台内容默认不得进入正式 Wiki。只有满足以下条件之一时才允许回流：

1. 新信息直接解释或修正冰冰小美原文中的概念、观点或推导；
2. 同一规律在多个标的或多个时间节点中得到重复验证；
3. 个股案例能清楚说明一个已存在的框架变量、风险传导或操作边界；
4. 用户明确要求把该案例沉淀为冰冰小美知识。

回流时只提炼可复用规律和必要案例证据，不复制整份研报、估值模型或逐日跟踪记录。优先更新已有 Concept、View、Reasoning、Timeline 或 Topic Page；新建页面前仍需检查重复。

## 9. 链接与索引边界

- 根 `index.md` 只导航正式 Wiki，不列出个股研报。
- `workbench/index.md` 统一导航标的、当前动作和下一验证点。
- 每个标的 `index.md` 导航其研报、跟踪、来源、模型和输出。
- Workbench 可以链接 Wiki 框架页，例如引用冰冰小美三要素或风险识别页面。
- Wiki 不反向链接每一个 Workbench 页面；只有被选为框架案例时才建立回链。
- 迁移时必须更新所有受影响的 Obsidian 路径链接，不能只移动文件。
- Obsidian 全局图谱默认过滤 `path:workbench`，需要查看工作台时再单独启用。

## 10. 规则文件调整

实施时需要同步修改：

- `AGENTS.md`：增加 Wiki 与 Workbench 的路由前置判断；
- `schema.md`：将正式知识层边界收紧为冰冰小美框架，并声明 Workbench 不属于八类正式页面；
- `page-types.md`：明确个股研报和动态跟踪不再默认使用 Query / Event / Timeline Page；
- `index.md`：删除个股研报导航，只保留回流后的框架知识；
- `log.md`：记录结构调整、迁移范围和未迁移项；
- 与个股研报生成有关的项目技能：把默认输出目录从 `wiki/queries/` 改为 `workbench/targets/`。

`workbench/AGENTS.md` 是工作台规则的唯一入口，避免把动态研究规则继续堆入正式 Wiki 的页面类型契约。

## 11. 现有内容迁移策略

本设计采用全量迁移现有14份机构级个股研报，避免只约束未来而保留当前边界污染。

迁移分四类处理：

1. **研报**：从 `wiki/queries/` 移到对应标的 `reports/`，正文结论不改写。
2. **单标的来源**：从根 `sources/` 移到标的 `sources/`；若同时被正式 Wiki 引用则保留原位。
3. **模型与输出**：从 `sources/automations/` 或其他输出位置迁入 `models/`、`outputs/`，但不移动与自动化运行状态强绑定的文件，除非同时修改自动化合同。
4. **反向链接**：更新 `index.md`、`hot.md`、Wiki 页面、来源页和工作台索引中的路径。

迁移必须使用版本控制可追踪的移动操作，不先删除再重建，不覆盖用户未提交修改。

## 12. 数据流

```text
公司公告 / 财报 / 行情 / 资金数据
        ↓
workbench/targets/<标的>/sources
        ↓
研报更新或 tracking 节点
        ↓
估值、风险状态和操作动作更新
        ↓
标的 index 与 workbench index
        ↓
满足回流门槛时，提炼规律到 wiki
```

## 13. 异常与边界处理

- 无法确认是否属于框架知识时，默认留在 Workbench，不进入 Wiki。
- 一个来源同时服务 Wiki 和 Workbench 时，优先保留根 `sources/` 的稳定路径，由 Workbench 链接该来源。
- 研报文件存在未提交修改时，迁移前先核对差异并保留工作区内容。
- 自动化脚本仍写入旧路径时，先修改并验证输出合同，再迁移对应文件。
- 迁移后发现断链时，不创建同名占位页；应修正真实链接目标。

## 14. 验收标准

实施完成后必须满足：

1. `workbench/` 与 `wiki/` 位于仓库根目录同级。
2. `wiki/queries/` 不再包含机构级个股决策研报。
3. 14份现有研报均能在对应标的目录找到，内容未丢失。
4. 根 `index.md` 不再导航个股研报，`workbench/index.md` 能导航全部标的。
5. 受迁移影响的内部链接不存在失效路径。
6. 个股研报生成技能默认写入 Workbench。
7. Workbench 工件不使用八类正式 Wiki 页面类型冒充知识页。
8. 中国中车等标的能够在 `tracking/` 中追加影响操作逻辑的日期节点。
9. 用户原有未提交修改保持完整，不被迁移覆盖或误提交。
10. 所有中文 Markdown 通过 UTF-8 和典型乱码字符检查。

## 15. 实施顺序

1. 建立 Workbench 目录、规则、模板和索引。
2. 修改根规则文件，建立明确路由契约。
3. 修改个股研究技能的默认输出位置。
4. 按标的逐个迁移研报及独占来源、模型和输出。
5. 更新全部反向链接与两个索引。
6. 配置图谱过滤并执行断链、编码、页面类型和文件数量验证。
7. 追加维护日志；仅提交本任务相关文件。
