# 页面类型规范

本文件定义 `llmwiki/wiki/` 下的页面类型。  
Agent 在整理资料、创建页面或重构页面时，必须根据资料性质选择合适的页面类型。

---

## 1. 页面类型总览

当前知识库使用以下七类页面：


| 页面类型           | 中文名   | 说明                  | 推荐目录              |
| -------------- | ----- | ------------------- | ----------------- |
| Concept Page   | 概念页   | 解释“是什么”             | `wiki/concepts/`  |
| Event Page     | 事件页   | 解释“发生了什么”           | `wiki/events/`    |
| View Page      | 观点页   | 解释“谁怎么看”            | `wiki/views/`     |
| Timeline Page  | 时间线页  | 解释“先后顺序是什么”         | `wiki/timelines/` |
| Reasoning Page | 推导链页  | 解释“为什么会这样推导”        | `wiki/reasoning/` |
| Topic Page     | 主题聚合页 | 组织一组相关页面, 承载“问题域入口” | `wiki/topics/`    |
| People Page    | 人物页   | 承载“作者入口”            | `wiki/people/`    |


核心理解：

```text
Concept Page   = 知识原子
People Page    = 观点和行动主体
Event Page     = 事实节点
View Page      = 主体观点
Timeline Page  = 事件序列
Reasoning Page = 因果推导
Topic Page     = 主题聚合页 / Hub 页
```

---

## 2. Concept Page：概念页

Concept Page 是稳定知识单元，用于解释概念、术语、模型、方法、框架。

它回答的问题是：

```text
这是什么？
它解决什么问题？
它由什么组成？
它和其他概念有什么关系？
```

适用于：

- 概念；
- 术语；
- 方法；
- 模型；
- 框架；
- 工具；
- 架构知识；
- 需求规格知识；
- 长期可复用的稳定知识。

示例：

```text
LLM Wiki
Obsidian-Wiki
Spec Kit
C4 Model
arc42
ADR
流动性
缩表
财政税收模式
风险节点
无风险收益率
```

推荐路径：

```text
wiki/concepts/
```

推荐结构：

```markdown
# 页面标题

## 定义

简要说明该概念是什么。

## 背景

说明该概念出现的背景、适用场景和问题来源。

## 核心观点

- 观点一
- 观点二
- 观点三

## 关键组成

说明该概念由哪些部分组成。

## 使用场景

说明它适合解决什么问题。

## 与其他概念的关系

- 与 [[概念A]] 的关系
- 与 [[概念B]] 的区别
- 与 [[概念C]] 的联系

## 常见误区

列出容易混淆或误用的地方。

## 相关页面

- [[相关页面A]]
- [[相关页面B]]

## 来源

- 来源一
- 来源二
```

## 3. People Page：人物页

本页面用于记录一个观点主体、投资者、作者、博主、研究者或关键人物的“人物画像”。

本页面重点关注：

- 这个人是谁；
- 这个人为什么值得进入知识库；
- 这个人的投资理念、价值观、性格、判断风格是什么；
- 这个人的观点适合如何使用；
- 这个人的观点有哪些边界和盲区。

本页面不负责承载全部文章、全部观点或全部推导。

推荐路径：

```text
wiki/people/
```

页面结构：

```markdown
# 人物名称

## 基本信息

本节重点关注：  
这个人的基础身份信息，以及他/她为什么出现在当前知识库中。不要在这里写具体观点细节。
应关注：

- 身份：
- 主要领域：
- 主要关注方向；
- 与当前知识库的关系；
- 是否是长期跟踪对象；
- 是否是某个体系、方法或观点的主要来源。

## 人物背景

本节重点关注：  
这个人的经历、位置、环境和长期形成判断方式的背景。

应关注：

- 这个人的职业背景；
- 所处市场、行业或知识领域；
- 长期关注的问题；
- 形成其判断风格的背景因素；
- 是否有明显的时代背景、市场背景或专业背景；
- 其背景如何影响观点来源和判断方式。

注意：  
人物背景不是百科介绍，只写和知识库相关的背景。

---

## 核心价值观
本节重点关注：
这个人相信什么、重视什么、排斥什么。
价值观是底层取舍，不是具体投资方法。
可以用一段话总结他用什么价值观和投资理念看市场？

## 投资理念

本节重点关注：
这个人如何把价值观落实到投资判断或现实判断中。

投资理念比价值观更具体，但还没有细到单次交易规则。

## 人物性格

本节重点关注：
这个人的判断气质和表达风格。

应关注：

是进攻型、防守型、怀疑型、乐观型、风险敏感型，还是产业链型；
是否倾向逆向思考；
是否容易提前提示风险；
是否重视现实约束；
是否对市场情绪敏感；
表达是温和、直接、反问式、强提醒式，还是系统分析式；
这种性格如何影响其观点的使用方式。

注意：
这里不是评价人品，而是刻画“判断风格”。

## 认知模式

本节重点关注：
这个人通常如何从信息推导出结论。

应关注：

他/她是从现象反推结构，还是从数据验证趋势；
他/她如何识别旧经验失效；
他/她如何看待市场共识；
他/她如何从短期事件推导长期变化；
他/她如何处理不确定性；
他/她是否有固定的因果推导模式；
他/她常用哪些问题来拆解市场。


## 风险偏好与仓位倾向
本节重点关注：
这个人面对风险和不确定性时的行动倾向。

应关注：

他/她是倾向进攻还是防守；
什么时候会建议观察；
什么时候会倾向等待；
什么时候会强调减仓或回避；
是否重视仓位管理；
如何看待左侧、右侧、追涨、补仓、重仓；
如何处理“判断不确定但机会存在”的场景。


```

## 4. Event Page：事件页

Event Page 用于记录单个事件、政策、会议、市场节点、公司动作、产业节点。

它回答的问题是：

```text
发生了什么？
什么时候发生？
谁参与？
背景是什么？
影响是什么？
它和哪些观点、推导、时间线有关？
```

适用于：

- 单个政策事件；
- 单个市场事件；
- 单个公司事件；
- 单个会议；
- 单个交易动作；
- 单个风险节点；
- 单个产业节点。

示例：

```text
2026-05-14-风险节点
2026-05-15-特朗普访华
大基金减持长电科技
美债收益率突破5%
```

推荐路径：

```text
wiki/events/
```

推荐命名：

```text
YYYY-MM-DD-事件标题.md
```

如果日期不确定，可以使用：

```text
事件标题.md
```

推荐结构：

```markdown
# 事件标题

## 事件概述

简要说明发生了什么。

## 时间

- 日期：
- 时间范围：
- 是否确定：

## 参与方

- [[人物A]]
- [[机构B]]
- [[公司C]]

## 背景

说明该事件发生前的背景。

## 影响

说明该事件造成或可能造成的影响。

## 相关观点

- [[观点页面A]]
- [[观点页面B]]

## 相关推导

- [[推导页面A]]
- [[推导页面B]]

## 所属时间线

- [[时间线A]]

## 相关概念

- [[概念A]]
- [[概念B]]

## 来源

- 来源一
- 来源二
```

## 5. View Page：观点页


View Page 用于记录一个阶段性观点、判断框架或投资结论。

它的核心职责不是记录事实本身，也不是展开完整推导链，而是回答：

> 在当前信息条件下，我怎么看这个问题？

View Page 关注的是“判断结果”和“观点演化”，尤其适合记录产业判断、投资观点、宏观判断、人物观点、市场风格判断等内容。

适用于：

- 某个作者对某个市场的看法；
- 某个博主对某个政策的判断；
- 某个机构对某个产业的观点；
- 某个人物对某个方法的主张；
- 用户自己对某个主题的阶段性判断。



推荐路径：

```text
wiki/views/
```

推荐命名：

```text
人物名对主题的看法.md
机构名对主题的判断.md
作者名关于主题的观点.md
```

推荐结构：

```markdown
# 观点名称

## 1. 当前核心观点

记录当前阶段的判断结论。

重点关注：
- 当前怎么看
- 判断方向是什么
- 适用范围是什么
- 观点是否仍然有效

---

## 2. 观点形成背景

记录这个观点最初是在什么背景下形成的。

重点关注：
- 当时发生了什么
- 为什么会产生这个判断
- 它试图解释什么问题

---

## 3. 观点时间线

记录观点如何随事件变化。

| 时间 / 阶段 | 事件节点 | 观点变化 | 变化原因 | 当前影响 |
|---|---|---|---|---|
| 阶段一 | 事件 A | 初始形成 | 原因 | 初步成立 |
| 阶段二 | 事件 B | 被强化 | 原因 | 置信度提升 |
| 阶段三 | 事件 C | 被修正 | 原因 | 边界变化 |

---

## 4. 判断依据摘要

用简短逻辑说明观点为什么成立。

注意：
- 只写摘要
- 不展开完整推导链
- 完整推导放到 Reasoning Page

相关推导：
- [[推导页面 A]]
- [[推导页面 B]]

---

## 5. 关键前提

记录观点成立所依赖的条件。

---

## 6. 反向证据与风险

记录什么情况发生后，这个观点可能失效。

---

## 7. 当前状态

观点状态：
- 初步形成
- 持续验证
- 被强化
- 被修正
- 部分失效
- 已证伪

---

## 8. 关联页面

相关 Event：
- [[]]

相关 Reasoning：
- [[]]

相关 Topic：
- [[]]

相关 People：
- [[]]

```

## 6. Timeline Page：时间线页

Timeline Page 用于记录多个事件的时间顺序。

注意：

```text
Timeline Page 只记录时间先后，不承担完整因果推导。
```

如果某个时间点背后有复杂因果逻辑，应链接到对应的 Reasoning Page。  
如果某个时间点来自某个作者观点，应链接到对应的 View Page。

适用于：

- 政策发布时间线；
- 市场事件时间线；
- 技术演进时间线；
- 项目阶段时间线；
- 产业发展时间线；
- 某个主题的发展脉络。

示例：

```text
市场事件时间线
AI Agent 发展时间线
半导体产业链事件时间线
财政税收模式转型时间线
```

推荐路径：

```text
wiki/timelines/
```

推荐结构：

```markdown
# 时间线标题

## 时间线

| 日期 | 事件 | 影响 | 相关页面 |
|---|---|---|---|
| YYYY-MM-DD | 事件描述 | 影响说明 | [[相关事件]] |

## 关键节点

说明哪些时间点是拐点、风险点或政策节点。

## 相关事件

- [[事件A]]
- [[事件B]]

## 相关观点

- [[观点页面A]]
- [[观点页面B]]

## 相关推导

- [[推导页面A]]
- [[推导页面B]]

## 相关概念

- [[概念A]]
- [[概念B]]

## 来源

- 来源一
- 来源二

## 7. Reasoning Page：推导链页

Reasoning Page 用于记录因果链、逻辑推导、观点拆解和风险演绎。

这类页面不是普通摘要，必须体现：

```text
前提 → 中间变量 → 传导机制 → 结论 → 风险点
```

适用于：

- 作者如何得出某个判断；
- 风险节点如何推导出来；
- 产业链逻辑；
- 宏观政策传导；
- 市场情绪与流动性变化；
- 事件之间的因果关系；
- 财政、产业、资产价格之间的传导。

生成规则引用

Reasoning Page 的具体生成要求，见：rules/reasoning-page-rules.md

推荐路径：

```text
wiki/reasoning/
```

推荐结构：

```markdown
# 页面标题

## 核心结论

用简洁语言说明作者或资料的主要结论。

## 推导前提

列出该结论依赖的前提。

- 前提一：
- 前提二：
- 前提三：

## 关键变量

列出推导中真正起作用的变量。

| 变量 | 含义 | 影响 |
|---|---|---|
| 变量A | 说明 | 影响路径 |
| 变量B | 说明 | 影响路径 |

## 推导链

## svg 推导图

如果 Reasoning Page 需要嵌入 SVG 推导图，必须使用 HTML `<img>` 标签，而不是 Markdown 图片语法。

推荐格式：

```html
<img src="../../sources/assets/example.svg"
     alt="图示说明"
     style="width: 90vw; max-width: none; height: auto;">
```

规则：

- `src` 使用从当前 wiki 页面到 `sources/assets/` 的相对路径；
- `alt` 使用页面标题或图示语义说明；
- `style` 固定为 `width: 90vw; max-width: none; height: auto;`，保证宽图在 Obsidian 中有足够展示宽度；
- 不使用 `![alt](path.svg)` 形式嵌入 SVG。

## 传导机制

说明变量之间为什么会发生传导。

例如：

- 流动性如何影响资产价格；
- 情绪如何影响杠杆；
- 政策如何影响产业资本；
- 财政压力如何影响地方投资；
- 算力中心如何影响城市税收结构。

## 时间节点

如果内容涉及时间，必须单独列出。


| 日期         | 事件  | 影响  |
| ---------- | --- | --- |
| YYYY-MM-DD | 事件  | 影响  |

列出什么情况会导致该推导成立或失效。

- 条件一：
- 条件二：
- 条件三：

## 反例与不确定性

必须说明该推导可能出错的地方。

- 反例一：
- 不确定性一：
- 待验证信息一：

## 8. Topic Page：主题聚合页 / Hub 页

Topic Page 用于组织一组相关页面。

它不是普通概念页，也不是文章摘要页。  
它的作用是做导航和聚合，组织一个问题域。

它回答的问题是：

```text
围绕这个主题，有哪些概念、人物、事件、观点、时间线、推导链和资料？
```

适用于：

- 市场风险推导；
- 半导体产业链；
- 财政税收模式转型；
- 儿童眼科就医；
- 任何需要长期跟踪的主题。

推荐路径：

```text
wiki/topics/
```

推荐结构：

```markdown
# 主题标题

## 主题说明

说明这个主题关注什么问题。

## 核心概念

- [[概念A]]
- [[概念B]]

## 关键人物

- [[人物A]]
- [[人物B]]

## 相关事件

- [[事件A]]
- [[事件B]]

## 相关观点

- [[观点页面A]]
- [[观点页面B]]

## 相关时间线

- [[时间线A]]
- [[时间线B]]

## 相关推导

- [[推导页面A]]
- [[推导页面B]]

## 重要资料

- 来源一
- 来源二

## 待补充问题

- 问题一
- 问题二
```

## 9. 页面类型选择规则

Agent 在整理资料时，应按以下规则判断页面类型。

---

### 9.1 默认不要直接创建 Topic Page

Topic Page 是聚合页，不是默认内容页。

默认优先判断资料是否属于：

1. Concept
2. People
3. Event
4. View
5. Timeline
6. Reasoning

只有当资料是在组织一个主题地图时，才创建或更新 Topic Page。

---

### 9.2 优先选择 Concept Page 的情况

当资料重点是解释一个稳定概念时，优先使用 Concept Page。

典型特征：

- 解释一个概念是什么；
- 说明方法、框架、模型；
- 可被多个页面复用；
- 不依赖单一事件或单篇文章。

---

### 9.3 优先选择 People Page 的情况

当资料重点是某个人或某个观点来源时，优先使用 People Page。

典型特征：

- 人物身份重要；
- 人物观点被多次引用；
- 人物参与了多个事件；
- 人物的行为影响观点或推导链；
- 需要追踪“谁说过什么”。

---

### 9.4 优先选择 Event Page 的情况

当资料重点是某个单独事件时，优先使用 Event Page。

典型特征：

- 有明确发生时间；
- 有明确参与方；
- 有事件结果或影响；
- 可被时间线引用；
- 可被观点或推导链引用。

---

### 9.5 优先选择 View Page 的情况

当资料重点是某个主体对某个问题的看法时，优先使用 View Page。

典型特征：

- 有明确观点来源；
- 有明确判断或立场；
- 重点是“谁怎么看”；
- 观点可能被后续推导引用；
- 不一定包含完整推导链。

---

### 9.6 优先选择 Timeline Page 的情况

当资料重点是多个事件的先后顺序时，优先使用 Timeline Page。

典型特征：

- 多个日期；
- 多个事件；
- 需要按时间排列；
- 不一定需要完整解释因果链。

---

### 9.7 优先选择 Reasoning Page 的情况

当资料重点是“作者如何得出某个判断”时，优先使用 Reasoning Page。

典型特征：

- 有明确结论；
- 有多个前提；
- 有因果传导；
- 有时间节点；
- 有风险触发条件；
- 有不确定性；
- 适合拆成“前提 → 变量 → 传导 → 结论”。

---

### 9.8 优先选择 Topic Page 的情况

当资料重点是组织某个领域的知识地图时，优先使用 Topic Page。

典型特征：

- 聚合多个概念；
- 聚合多个人物；
- 聚合多个事件；
- 聚合多个观点；
- 聚合多条推导；
- 作为导航入口；
- 不是单篇资料的摘要。

---

## 10. 页面类型组合规则

有些资料可能同时包含多种内容。此时应拆分处理。

---

### 10.1 同时有概念和观点

处理方式：

- 稳定概念写入 Concept Page；
- 某个主体对该概念的看法写入 View Page；
- View Page 链接 Concept Page。

示例：

```markdown
相关概念：[[LLM Wiki]]
相关观点：[[Karpathy对LLM Wiki的观点]]
```

---

### 10.2 同时有概念和推导

处理方式：

- 稳定概念写入 Concept Page；
- 当前文章中的因果分析写入 Reasoning Page；
- Reasoning Page 链接 Concept Page。

示例：

```markdown
相关概念：[[流动性]]、[[市场情绪]]
相关推导：[[冰冰小美-5月14日风险节点推导]]
```

---

### 10.3 同时有人物和观点

处理方式：

- 人物身份和稳定信息写入 People Page；
- 具体观点写入 View Page；
- People Page 与 View Page 互相链接。

示例：

```markdown
观点来源：[[Michael Burry]]
相关观点：[[Michael Burry对半导体做空的判断]]
```

---

### 10.4 同时有观点和推导

处理方式：

- 观点本身写入 View Page；
- 观点背后的推导链写入 Reasoning Page；
- 两个页面互相链接。

示例：

```markdown
相关观点：[[蓝庭集对有色金属的看法]]
相关推导：[[冰冰小美-有色金属拐点推导]]
```

---

### 10.5 同时有事件和时间线

处理方式：

- 单个事件写入 Event Page；
- 多个事件顺序写入 Timeline Page；
- Event Page 与 Timeline Page 互相链接。

示例：

```markdown
所属时间线：[[市场事件时间线]]
相关事件：[[2026-05-14-风险节点]]
```

---

### 10.6 同时有时间线和推导链

处理方式：

- 时间顺序写入 Timeline Page；
- 因果分析写入 Reasoning Page；
- 两个页面互相链接。

示例：

```markdown
相关时间线：[[市场事件时间线]]
相关推导：[[冰冰小美-5月14日风险节点推导]]
```

---

### 10.7 同时有主题和具体页面

处理方式：

- 具体知识写入 Concept / People / Event / View / Timeline / Reasoning；
- Topic Page 只负责聚合链接；
- 不要把所有内容都堆进 Topic Page。

示例：

```markdown
所属主题：[[市场风险推导]]
相关概念：[[流动性]]
相关观点：[[某作者对市场风险的判断]]
相关事件：[[2026-05-14-风险节点]]
相关推导：[[冰冰小美-5月14日风险节点推导]]
```

---

## 11. 页面创建原则

Agent 创建新页面前必须检查：

1. 是否已有同义页面；
2. 是否可以更新已有页面；
3. 新页面是否属于稳定知识单元；
4. 新页面是否能被未来复用；
5. 新页面属于哪种页面类型；
6. 新页面是否需要加入 `index.md`；
7. 新页面是否需要与现有页面建立双链。

如果只是临时摘录，不应创建新的 wiki 页面，应放入 `sources/`。

---

## 12. 页面质量要求

每个 `wiki/` 页面至少应满足：

1. 有清晰标题；
2. 有明确页面类型；
3. 有来源；
4. 有相关页面；
5. 有必要的双链；
6. 没有把观点误写成事实；
7. 没有把推测误写成结论；
8. 没有与其他页面重复表达同一概念、人物、事件或观点；
9. 没有把 Topic Page 当成普通笔记页；
10. 没有把 View Page 和 Reasoning Page 混用；
11. 没有把 Timeline Page 和 Reasoning Page 混用。
