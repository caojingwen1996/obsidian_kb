# 页面类型规范

本文件定义 `llmwiki/wiki/` 下的页面类型。  
Agent 在整理资料、创建页面或重构页面时，必须根据资料性质选择合适的页面类型。

各页面类型的可复制模板已拆分到 `templates/` 目录下。本文保留页面类型语义、适用场景和选择规则；创建新页面时必须以对应模板作为输出结构，并按资料内容补齐。

---

## 1. 页面类型总览

当前知识库使用以下八类页面：


| 页面类型           | 中文名   | 说明                  | 推荐目录              |
| -------------- | ----- | ------------------- | ----------------- |
| Concept Page   | 概念页   | 解释“是什么”             | `wiki/concepts/`  |
| Event Page     | 事件页   | 解释“发生了什么”           | `wiki/events/`    |
| View Page      | 观点页   | 解释“谁怎么看”            | `wiki/views/`     |
| Timeline Page  | 时间线页  | 解释“先后顺序是什么”         | `wiki/timelines/` |
| Reasoning Page | 推导链页  | 解释“为什么会这样推导”        | `wiki/reasoning/` |
| Query Page     | 问题页   | 解释“这个问题当前如何回答”      | `wiki/queries/`   |
| Topic Page     | 主题聚合页 | 组织一组相关页面, 承载“问题域入口” | `wiki/topics/`    |
| People Page    | 人物页   | 承载“作者入口”            | `wiki/people/`    |


核心理解：

```text
Concept Page   = 知识原子
People Page    = 观点和行动主体
Event Page     = 事实节点
View Page      = 主体观点
Timeline Page  = 事件序列
Reasoning Page = 因果推导
Query Page     = 问题驱动的回答沉淀
Topic Page     = 主题聚合页 / Hub 页
```

## 2. Concept Page：概念页

Concept Page 是稳定知识单元，用于解释概念、术语、模型、方法、框架。

它回答的问题是：

```text
这是什么？
它解决什么问题？
它由什么组成？
它和其他概念有什么关系？
```

适用于：

- 概念；
- 术语；
- 方法；
- 模型；
- 框架；
- 工具；
- 架构知识；
- 需求规格知识；
- 长期可复用的稳定知识。

Concept 页面写作风格要求：

Concept 页面用于解释一个概念、框架或作者体系中的专有说法。写作时要保持“定义清楚、来源明确、边界克制、推导不过度”。

生成规则引用：

Concept Page 的具体生成要求，见：`rules/concept-page-rules.md`

推荐路径：

```text
wiki/concepts/
```

必须参考的输出模板：

- `templates/concept-page-template.md`

---

## 3. People Page：人物页

People Page 是人物画像入口，不承载该人物的全部文章、全部来源和全部相关页面，负责记录人物定位、价值观、判断理念、认知模式、方法论体系、代表观点、代表推导和使用边界。

### 页面职责

生成 People Page 时，必须优先回答：

- 这个人是谁；
- 为什么值得跟踪；
- 这个人相信什么；
- 这个人如何判断问题；
- 这个人的核心方法论是什么；
- 这个人适合在哪些场景被调用；
- 这个人的观点有什么使用边界。

### 内容边界

People Page 不承载全部文章、全部来源、全部观点、全部推导和全部相关页面。

单篇观点进入 View Page。  
完整推导进入 Reasoning Page。  
时间演变进入 Timeline Page。  
抽象概念进入 Concept Page。  
具体事件进入 Event Page。  
大量文章、观点、推导、来源索引进入 Topic 地图页。

### related 字段规则

`related` 只放核心入口，建议 8—15 个，最多不超过 20 个。

优先放：

- 核心 Topic Page；
- 核心 Concept Page；
- 核心 Timeline Page；
- 知识地图页；
- 少量最关键 Reasoning Page。

禁止把全部 View Page、全部 Reasoning Page、全部 Source Page、全部普通 Concept Page 塞进 `related`。

### sources 字段规则

People Page 不直接堆放大量 source。

如果来源很多，创建：

`wiki/topics/人物名-知识地图.md`

并在 People Page 中使用：

`source_index: "[[topics/人物名-知识地图#来源索引]]"`

### 代表观点规则

代表观点只索引 View Page。

数量建议 5—8 个，最多不超过 12 个。  
每个链接后写一句话说明它如何体现人物判断。

### 代表推导规则

代表推导只索引 Reasoning Page。

数量建议 5—8 个，最多不超过 12 个。  
每个链接后写一句话说明它如何体现人物推导方式。

### 长文压缩规则

People Page 中每个方法论模块只写总览。

完整因果链放到 Reasoning Page。  
单篇作者观点放到 View Page。  
观点演变放到 Timeline Page。  
全部文章和来源放到 Topic 地图页。

### 事实、推断、待验证分离

整理人物画像时，必须区分：

- 明确事实；
- 作者原文观点；
- 我的复述；
- 我的推断；
- 待验证判断。

不能把一次文章、一次交割单或一次发言直接上升为稳定人格特征。  
如果证据不足，标注“暂定”。

### 最终检查

生成 People Page 后，检查以下问题：

- 页面是否能快速说明这个人是谁；
- related 是否超过 20 个；
- sources 是否堆了大量原文；
- 代表观点是否只放 View Page；
- 代表推导是否只放 Reasoning Page；
- 是否出现大量无说明链接；
- 是否把文章索引误放进人物页；
- 是否保留了人物使用边界和可能盲区。

推荐路径：

```text
wiki/people/
```

### 输出模板：

- `templates/people-page-template.md`

## 4. Event Page：事件页

Event Page 用于记录单个事件、政策、会议、市场节点、公司动作、产业节点。

它回答的问题是：

```text
发生了什么？
什么时候发生？
谁参与？
背景是什么？
影响是什么？
它和哪些观点、推导、时间线有关？
```

适用于：

- 单个政策事件；
- 单个市场事件；
- 单个公司事件；
- 单个会议；
- 单个交易动作；
- 单个风险节点；
- 单个产业节点。

示例：

```text
2026-05-14-风险节点
2026-05-15-特朗普访华
大基金减持长电科技
美债收益率突破5%
```

推荐路径：

```text
wiki/events/
```

推荐命名：

```text
YYYY-MM-DD-事件标题.md
```

如果日期不确定，可以使用：

```text
事件标题.md
```

必须参考的输出模板：

- `templates/event-page-template.md`

## 5. View Page：观点页
View Page 用于记录“谁，在什么背景下，怎么看这个问题”。

它重点承载作者、机构或用户对某个问题的阶段性判断，包括判断结果、判断背景、观点边界、适用条件和观点演化。

View Page 不负责保存事实原文，不负责完整展开推导链，也不负责定义抽象概念。

它的核心问题是：

谁，在什么背景下，怎么看这个问题？

生成逻辑：如果view_type分类表中定义了rule,

推荐路径：

```text
wiki/views/
```

### view_type分类表
1. policy_interpretation：政策解读型观点页

用于记录作者如何解读一项政策、会议、制度、改革、监管变化或财政货币工具。

type: view
view_type: policy_interpretation

适用于：

政策解读；
监管变化判断；
财政政策判断；
货币政策判断；
制度改革判断；
政策对市场或产业的影响判断。

核心问题：

作者如何理解这项政策？它要解决什么问题？它通过什么路径影响市场、产业或资产价格？

具体生成逻辑见：rules/policy-view-page-rules.md

2. macro_judgment：宏观判断型观点页

用于记录作者对外部货币环境、全球风险、大国关系、世界格局、国运位置和长期市场方向的判断。

type: view
view_type: macro_judgment

适用于：

大国关系判断；
国运与国家位置判断；
经济周期判断；
长期牛熊判断；
世界格局变化判断；
宏观叙事对资本市场的影响判断。

核心问题：

作者如何判断当前所处的宏观阶段？这个阶段会如何影响市场方向、资产定价和投资主线？

3. liquidity_judgment：流动性判断型观点页

用于记录作者对央行、利率、信用、银行体系、长期资金、市场资金结构和流动性环境的判断。

type: view
view_type: liquidity_judgment

适用于：

央行政策判断；
利率变化判断；
信用环境判断；
银行体系资金判断；
长期资金入市判断；
股债性价比判断；
流动性对市场风格的影响判断。

核心问题：

作者如何判断钱的方向、钱的价格和钱的约束？流动性变化会如何影响市场估值和资产偏好？

4. industry_judgment：产业判断型观点页

用于记录作者对某个产业趋势、供需格局、竞争格局、技术路线、产业链变化和企业受益方向的判断。

type: view
view_type: industry_judgment

适用于：

产业趋势判断；
供需变化判断；
技术路线判断；
竞争格局判断；
产业链重构判断；
政策对产业影响判断；
产业中受益环节判断。

核心问题：

作者如何判断这个产业的变化？变化来自需求、供给、政策、技术，还是竞争格局？哪些环节会受益
---

## 6. Timeline Page：时间线页

Timeline Page 用于记录一个主题在时间维度上的阶段演进、关键节点、节奏变化和验证窗口。
它的核心作用是：  
把分散事件放回同一条时间轴中，帮助我们看清「某个观点、产业、政策、市场情绪或投资主线」是如何一步步演化的。

注意：

```text
Timeline Page 只记录时间先后，不承担完整因果推导。
```

如果某个时间点背后有复杂因果逻辑，应链接到对应的 Reasoning Page。  
如果某个时间点来自某个作者观点，应链接到对应的 View Page。

### 页面职责

Timeline Page 主要承担以下职责：

1. **记录时间顺序**
  按照时间先后记录事件、政策、行情、产业变化、作者观点和交割单节点。
2. **标记阶段位置**
  每个时间点都要尽量标注它属于哪个阶段，例如：
  - 酝酿期；
  - 启动期；
  - 加速期；
  - 分化期；
  - 兑现期；
  - 透支期；
  - 泡沫期；
  - 退潮期；
  - 再验证期。
3. **沉淀关键时间锚**
  对作者反复提到的年份、月份或阶段，要单独标注其意义。
   例如：
  - 2024：某轮政策或行情起点；
  - 2026：十五五开局、AI/半导体预期交易阶段；
  - 2028：AI/半导体兑现窗口、中美科技竞争观察点、估值透支检验点；
  - 2030：十五五收官、长期产业规划验收点。
4. **连接事件与推导**
  时间线页只负责放置节点和阶段判断。  
   如果某个节点背后存在复杂因果链，应链接到对应的 Reasoning Page。
5. **连接作者观点**
  如果某个时间点来自某位作者的判断、预测或阶段划分，应链接到对应的 View Page 或 People Page。
6. **提供复盘入口**
  Timeline Page 要能帮助后续复盘：
  - 当时作者怎么看；
  - 当时市场在交易什么；
  - 后续是否被验证；
  - 哪些预期被提前透支；
  - 哪些节点成为真正拐点；
  - 哪些节点只是短期情绪。

推荐路径：

```text
wiki/timelines/
```

必须参考的输出模板：

- `templates/timeline-page-template.md`

## 7. Reasoning Page：推导链页

Reasoning Page 用于记录因果链、逻辑推导、观点拆解和风险演绎。

这类页面不是普通摘要，必须体现：

```text
前提 → 中间变量 → 传导机制 → 结论 → 风险点
```

适用于：

- 作者如何得出某个判断；
- 风险节点如何推导出来；
- 产业链逻辑；
- 宏观政策传导；
- 市场情绪与流动性变化；
- 事件之间的因果关系；
- 财政、产业、资产价格之间的传导。

生成规则引用

Reasoning Page 的具体生成要求，见：rules/reasoning-page-rules.md

推荐路径：

```text
wiki/reasoning/
```

必须参考的输出模板：

- `templates/reasoning-page-template.md`

## 8. Topic Page：主题聚合页 / Hub 页

Topic Page 用于承载一个长期主题、问题域、内容地图或体系模块。

它的核心职责是回答：

> 这个主题是什么？它包含哪些子问题？相关观点、推导、概念、事件、时间线和来源如何组织在一起？

Topic Page 是知识库中的主题中枢。  
它聚合多个 View Page、Reasoning Page、Concept Page、Timeline Page、Event Page 和 Source Page，让一个长期主题具备清晰入口、边界、结构和后续维护路径。

Topic Page 的结构可以随主题类型变化，但必须遵守统一的底层规范。

### Topic Page 的职责边界

Topic Page 负责：

- 聚合主题；
- 定义范围；
- 组织子问题；
- 连接相关页面；
- 标记当前阶段；
- 沉淀主题地图；
- 提示后续验证点。

推荐路径：

```text
wiki/topics/
```

必须参考的输出模板：

- `templates/topic-page-template.md`

## 9. Query Page：问题页 / 问答沉淀页

Query Page 用于记录一个真实提出过的问题，以及当前基于知识库得到的回答、推导逻辑、相关页面和后续归档状态。

它不是主题聚合页，也不是单篇资料摘要。
它的作用是把“为什么 / 如何理解 / 当前怎么看”这类问题沉淀为可复用的问答入口。

它回答的问题是：

```text
原始问题是什么？
当前核心答案是什么？
答案依据哪些页面或资料？
这个问题是否已经补充到正式页面？
```

适用于：

- 用户提出的高复用问题；
- 一次 wiki 查询后的综合回答；
- 需要保留问题语境的学习记录；
- 暂时还不适合拆成 Concept / View / Reasoning / Topic 的中间问题；
- 已回答但需要后续补入正式页面的问题。

推荐路径：

```text
wiki/queries/
```

必须参考的输出模板：

- `templates/query-page-templates.md`

## 10. 页面类型选择规则

Agent 在整理资料时，应按以下规则判断页面类型。

---

### 10.1 默认不要直接创建 Topic Page

Topic Page 是聚合页，不是默认内容页。

默认优先判断资料是否属于：

1. Concept
2. People
3. Event
4. View
5. Timeline
6. Reasoning
7. Query

只有当资料是在保留可复用问题的回答语境时，才创建 Query Page。
只有当资料是在组织一个主题地图时，才创建或更新 Topic Page。

---

### 10.2 优先选择 Concept Page 的情况

当资料重点是解释一个稳定概念时，优先使用 Concept Page。

典型特征：

- 解释一个概念是什么；
- 说明方法、框架、模型；
- 可被多个页面复用；
- 不依赖单一事件或单篇文章。

---

### 10.3 优先选择 People Page 的情况

当资料重点是某个人或某个观点来源时，优先使用 People Page。

典型特征：

- 人物身份重要；
- 人物观点被多次引用；
- 人物参与了多个事件；
- 人物的行为影响观点或推导链；
- 需要追踪“谁说过什么”。

---

### 10.4 优先选择 Event Page 的情况

当资料重点是某个单独事件时，优先使用 Event Page。

典型特征：

- 有明确发生时间；
- 有明确参与方；
- 有事件结果或影响；
- 可被时间线引用；
- 可被观点或推导链引用。

---

### 10.5 优先选择 View Page 的情况

当资料重点是某个主体对某个问题的看法时，优先使用 View Page。

典型特征：

- 有明确观点来源；
- 有明确判断或立场；
- 重点是“谁怎么看”；
- 观点可能被后续推导引用；
- 不一定包含完整推导链。

---

### 10.6 优先选择 Timeline Page 的情况

当资料重点是多个事件的先后顺序时，优先使用 Timeline Page。

典型特征：

- 多个日期；
- 多个事件；
- 需要按时间排列；
- 不一定需要完整解释因果链。

---

### 10.7 优先选择 Reasoning Page 的情况

当资料重点是“作者如何得出某个判断”时，优先使用 Reasoning Page。

典型特征：

- 有明确结论；
- 有多个前提；
- 有因果传导；
- 有时间节点；
- 有风险触发条件；
- 有不确定性；
- 适合拆成“前提 → 变量 → 传导 → 结论”。

---

### 10.8 优先选择 Query Page 的情况

当资料重点是保留一个问题的当前回答和查询语境时，优先使用 Query Page。

典型特征：

- 有明确的原始问题；
- 回答综合了多个页面或资料；
- 问题本身具有复用价值；
- 需要记录“已得到初步答案 / 已整理推导逻辑 / 已补充到正式页面”等状态；
- 暂时不适合直接拆成 Concept、View、Reasoning 或 Topic。

---

### 10.9 优先选择 Topic Page 的情况

当资料重点是组织某个领域的知识地图时，优先使用 Topic Page。

典型特征：

- 聚合多个概念；
- 聚合多个人物；
- 聚合多个事件；
- 聚合多个观点；
- 聚合多条推导；
- 聚合多个问题页；
- 作为导航入口；
- 不是单篇资料的摘要。

---

## 11. 页面类型组合规则

有些资料可能同时包含多种内容。此时应拆分处理。

---

### 11.1 同时有概念和观点

处理方式：

- 稳定概念写入 Concept Page；
- 某个主体对该概念的看法写入 View Page；
- View Page 链接 Concept Page。

示例：

```markdown
相关概念：[[LLM Wiki]]
相关观点：[[Karpathy对LLM Wiki的观点]]
```

---

### 11.2 同时有概念和推导

处理方式：

- 稳定概念写入 Concept Page；
- 当前文章中的因果分析写入 Reasoning Page；
- Reasoning Page 链接 Concept Page。

示例：

```markdown
相关概念：[[流动性]]、[[市场情绪]]
相关推导：[[冰冰小美-5月14日风险节点推导]]
```

---

### 11.3 同时有人物和观点

处理方式：

- 人物身份和稳定信息写入 People Page；
- 具体观点写入 View Page；
- People Page 与 View Page 互相链接。

示例：

```markdown
观点来源：[[Michael Burry]]
相关观点：[[Michael Burry对半导体做空的判断]]
```

---

### 11.4 同时有观点和推导

处理方式：

- 观点本身写入 View Page；
- 观点背后的推导链写入 Reasoning Page；
- 两个页面互相链接。

示例：

```markdown
相关观点：[[蓝庭集对有色金属的看法]]
相关推导：[[冰冰小美-有色金属拐点推导]]
```

---

### 11.5 同时有事件和时间线

处理方式：

- 单个事件写入 Event Page；
- 多个事件顺序写入 Timeline Page；
- Event Page 与 Timeline Page 互相链接。

示例：

```markdown
所属时间线：[[市场事件时间线]]
相关事件：[[2026-05-14-风险节点]]
```

---

### 11.6 同时有时间线和推导链

处理方式：

- 时间顺序写入 Timeline Page；
- 因果分析写入 Reasoning Page；
- 两个页面互相链接。

示例：

```markdown
相关时间线：[[市场事件时间线]]
相关推导：[[冰冰小美-5月14日风险节点推导]]
```

---

### 11.7 同时有主题和具体页面

处理方式：

- 具体知识写入 Concept / People / Event / View / Timeline / Reasoning / Query；
- Topic Page 只负责聚合链接；
- 不要把所有内容都堆进 Topic Page。

示例：

```markdown
所属主题：[[市场风险推导]]
相关概念：[[流动性]]
相关观点：[[某作者对市场风险的判断]]
相关事件：[[2026-05-14-风险节点]]
相关推导：[[冰冰小美-5月14日风险节点推导]]
```

## 12. 文件命名规则

页面命名应清晰、稳定、可复用。

### 12.1 通用命名

优先使用：

```text
中文概念名.md
英文术语.md
中文说明 + 英文术语.md
```

避免：

```text
新建文档.md
未命名.md
总结1.md
文章整理.md
临时.md
```

同一概念、人物、事件、观点不得重复建立多个页面。

### 12.2 Concept Page 命名

Concept Page 文件名建议使用：

```text
人物名-页面用途-核心对象.md
```

其中：

- `人物名`：概念、规则、模型或方法的主要来源人物。
- `页面用途`：说明页面用途或知识单元类型，例如 `rule`、`framework`、`model`、`concept`。
- `核心对象`：说明页面沉淀的核心概念、规则对象或方法对象。

页面用途类型列表：


| 页面用途        | 含义             |
| ----------- | -------------- |
| `rule`      | 执行参考、判断规则或操作规则 |
| `framework` | 包含多个要素的一整套体系   |
| `concept`   | 独立的原子概念        |


示例：

```text
碧树西风-rule-交易结构拆解.md
```

如果概念不是来自某个明确人物，而是公共概念或通用术语，可以继续使用稳定的中文概念名。

### 12.3 Event Page 命名

事件页建议使用：

```text
YYYY-MM-DD-事件标题.md
```

示例：

```text
2026-05-14-风险节点.md
2026-05-15-特朗普访华.md
```

如果日期不确定，可以使用：

```text
事件标题.md
```

### 12.4 People Page 命名

人物页建议使用人物常用名。

示例：

```text
Andrej Karpathy.md
Michael Burry.md
特朗普.md
蓝庭集.md
```

如果同名人物可能混淆，可在标题中加身份说明。

### 12.5 View Page 命名

人物名-页面用途-核心对象.md

示例：

```text
冰冰小美-政策解读-中央加杠杆.md.md
```

## 13. 页面创建原则

Agent 创建新页面前必须检查：

1. 是否已有同义页面；
2. 是否可以更新已有页面；
3. 新页面是否属于稳定知识单元；
4. 新页面是否能被未来复用；
5. 新页面属于哪种页面类型；
6. 新页面是否需要加入 `index.md`；
7. 新页面是否需要与现有页面建立双链。

如果只是临时摘录，不应创建新的 wiki 页面，应放入 `sources/`。

### 13.1 页面创建阈值

只有满足以下条件之一，才应创建新的正式 `wiki/` 页面：

1. 某个概念、框架或变量具有长期复用价值。
2. 某个人物值得持续跟踪，且其观点会被多次引用。
3. 某个事件是事实节点，可被观点、时间线或推导链引用。
4. 某条观点有明确主体、来源、时间或上下文，并能服务现有主题。
5. 某个推导链包含明确前提、变量、传导机制、结论和不确定性。
6. 某个主题需要聚合多个概念、人物、事件、观点或推导链。
7. 某条时间线能显著提升对多个事件先后关系的理解。
8. 某个问题具有复用价值，需要保留原始问题、当前答案、推导逻辑和归档状态。

以下情况不要新建正式页面：

- 内容过于零碎、临时或一次性。
- 可以补入已有页面而不损害结构清晰度。
- 只是原始摘录，尚未形成稳定知识。
- 只能制造孤立页面，无法接入现有双链。
- 来源不足，无法区分事实、观点和推测。

---

## 14. 页面质量要求

每个 `wiki/` 页面至少应满足：

1. 有清晰标题；
2. 有明确页面类型；
3. 有来源；
4. 有相关页面；
5. 有必要的双链；
6. 没有把观点误写成事实；
7. 没有把推测误写成结论；
8. 没有与其他页面重复表达同一概念、人物、事件或观点；
9. 没有把 Topic Page 当成普通笔记页；
10. 没有把 Query Page 当成所有问答的默认归宿；
11. 没有把 View Page 和 Reasoning Page 混用；
12. 没有把 Timeline Page 和 Reasoning Page 混用。
