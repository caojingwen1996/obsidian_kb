# 产业分析模型 Skill

## 1. Skill 定位

本 Skill 是产业分析的**执行引擎**。

它负责：

- 识别何时需要启动产业分析；
- 每次执行前重新读取最新产业分析框架；
- 将知识库内容转换为本次任务可执行的分析结构；
- 规划和搜集所需证据；
- 按最新框架逐模块完成分析；
- 处理模块之间的传导、冲突和阶段错位；
- 调用 `template.md` 生成最终输出；
- 将完整报告保存为 Markdown 母稿，并生成 HTML 阅读版放入对应产业目录；
- 执行质量检查并披露不确定性。

本 Skill 不负责维护产业分析框架的具体内容。

以下内容只能在知识库主文件中维护：

- 一级模块及顺序；
- 子模块及定义；
- 阶段名称和阶段划分；
- 各模块核心问题；
- 指标体系；
- 判断规则；
- 概念命名；
- 方法论更新。

知识库主文件：

`wiki/concepts/冰冰小美-framework-产业思维.md`

该文件是产业分析框架的唯一知识源和权威来源。
执行时只采用该文件当前定义的模块、顺序、术语、判断标准和行动分类；不得在本技能中缓存、复制或补充框架内容。

---

## 2. 文件职责与优先级

### 2.1 知识库主文件

职责：

- 定义“分析什么”；
- 定义模块、阶段、概念和判断规则；
- 定义当前有效的产业分析框架；
- 承载框架迭代。

### 2.2 skill.md

职责：

- 定义“如何执行”；
- 读取和解析最新框架；
- 组织证据；
- 完成分析；
- 做一致性检查；
- 调用输出模板。

### 2.3 template.md

职责：

- 定义“如何展示”；
- 提供完整产业分析报告的展示结构；
- 本 Skill 只允许调用“完整产业分析报告”，不得选择、压缩或自行改写为其他输出版本；
- 不决定产业框架内容。

### 2.4 冲突优先级

出现冲突时，优先级固定为：

```text
知识库中的最新框架定义
>
skill.md 中的执行规则
>
template.md 中的展示结构
```

处理原则：

1. 知识库新增、删除、改名或调整模块时，按最新知识库执行；
2. 模板缺少对应模块时，使用通用模块结构临时渲染；
3. 不得为了适配旧模板而忽略知识库中的新内容；
4. 模板中存在知识库已删除的模块时，不再输出该模块；
5. 输出末尾记录本次使用的框架版本与模板适配情况。

## 3. 触发条件

以下请求应启动本 Skill：

- 分析某个产业、行业、赛道或产业主题；
- 解读产业规划、行业增加值、投资增速、产量、销量、库存、价格或行业利润；
- 拆解产业链、利润池、瓶颈、增量来源或价值迁移；
- 判断产业发展阶段、供需周期、资本开支周期、利润周期或投资生命周期；
- 判断一个产业处于扩张、供给释放、泡沫、出清或再成长阶段；
- 分析政策如何传导到项目、投资、产能、订单和利润；
- 从宏观数据或政策信号寻找产业增量。

以下任务应转入其他流程：

- 单家公司完整基本面分析；
- 单家公司治理、管理层和组织能力分析；
- 单家公司财务建模；
- 单家公司估值和投资建议；
- 纯宏观经济分析；
- 只查询一个数据且无需形成产业判断。

企业数据可以用于验证产业结构、供需、扩产和竞争格局，但不得将产业分析写成单家公司研究报告。

## 4. 每次执行的强制加载规则

每次执行产业分析任务时，必须完整读取当前版本：

`wiki/concepts/冰冰小美-framework-产业思维.md`

不得使用以下替代品：

- 历史对话中的框架摘要；
- 之前生成的产业报告；
- 模型记忆中的模块名称；
- 旧版 `skill.md` 中曾出现的框架结构；
- 缓存的阶段划分；
- `template.md` 中写死的章节。

### 4.1 必须提取的框架信息

读取知识库后，形成“本次运行框架快照”，至少提取：

- 框架名称；
- 框架 ID；
- 框架版本；
- 更新时间；
- 框架状态；
- 一级模块名称与顺序；
- 每个模块的目标；
- 子模块结构；
- 每个模块的必答问题；
- 当前有效的阶段名称；
- 各阶段的定义和判断条件；
- 建议使用的指标；
- 模块之间的传导关系；
- 质量检查规则；
- 框架中特别强调的边界或禁区。

### 4.2 框架元数据缺失时

若知识库未提供版本号或结构化元数据：

1. 仍需读取全文；
2. 根据标题层级和正文提取当前结构；
3. 将框架版本标记为“未声明”；
4. 将解析方式标记为“根据正文推导”；
5. 不得自行沿用历史版本号；
6. 在最终输出中提示知识库缺少明确版本标识。

### 4.3 框架无法读取时

若主知识库文件无法访问、为空或严重损坏：

- 停止正式分析；
- 说明无法加载框架；
- 不得凭记忆继续生成完整产业报告；
- 可以列出已获得的数据，但必须标注“尚未按知识库框架完成分析”。

## 6. 输入解析

收到任务后，先识别以下信息：

- 产业名称；
- 分析对象的统计口径；
- 地域范围；
- 时间范围；
- 数据截止日期；
- 用户核心问题；
- 是否需要资本市场视角；
- 是否需要候选产业环节；
- 用户是否提供了政策、数据、文件或已有结论。

### 6.1 信息不完整时

能够通过可靠资料补全的内容，直接补全。

以下情况需要明确提示：

- 产业边界存在多种定义；
- 用户给出的行业数据口径过宽；
- 缺少关键统计数据；
- 规划目标与实际执行数据无法匹配；
- 某些判断只能通过代表性企业旁证；
- 最新信息尚未公开。

## 7. 唯一输出契约

所有触发本 Skill 的任务，无论用户要求快速判断、单一模块分析、日报周报、阶段监测还是多产业比较，都必须：

1. 重新加载完整产业框架；
2. 完成框架要求的全部模块分析；
3. 严格按 `template.md` 中“完整产业分析报告”的章节、顺序和表格结构输出；
4. 保留完整报告的 `0—7` 章以及“通用风险与证据附录”；
5. 对非用户重点但证据不足的章节标记“待验证”，不得删除章节；
6. 不得选择简版、专项版、监测版、比较版或任何自行压缩的输出形式。
7. 在“产业结构”中完整输出“产业链公司映射”，覆盖上游、中游、下游的主要子环节；逐家公司或主体列出产业链位置、核心产品、上市属性 / 证券代码、业务占比或纯度、证据状态和来源时间。
8. 分析完成后必须保存完整 Markdown 母稿，并生成 HTML 阅读版；两份文件使用相同文件名主体，放入同一个对应产业目录。

用户问题只决定报告的研究重点和证据投入，不改变完整报告的输出结构。
每次运行均须保存 Markdown 和 HTML 产物，但不得因此自动创建或改写正式 `wiki/` 页面；只有用户明确要求沉淀为正式知识页时，才进入 Wiki 维护流程。

### 7.1 产业链公司映射强制口径

产业链公司映射用于验证价值量、瓶颈和订单流向，不是罗列概念股。每次报告必须遵守：

- 上游、中游、下游均需列出代表公司或主体；某层没有可核验主体时写“未获取到”，不得留空或删除该层。
- 优先列出直接解决产业约束、拥有真实订单或承担实际任务的主体；兼顾未上市公司、科研院所和上市公司，不能只列资本市场标的。
- “业务占比”只填写公司法定披露、交易所文件或可复核分部数据中的比例；需要写明统计年度和分母口径。
- 公司未披露精确比例时，写“未单独披露”，再使用“高 / 中高 / 中 / 中低 / 低”表达业务纯度，并说明判断依据。不得把定性纯度伪装成收入比例。
- 纯度判断只描述目标产业对公司收入、订单、资产或战略的相关程度，不替代公司估值与投资建议。
- 证据状态使用“已披露 / 部分披露 / 未单独披露 / 待验证”；任务、订单、产品能力和收入确认必须分别标注。
- 不得把集团任务直接映射为上市公司收入，不得把产品能力、合作关系或概念关联写成已取得订单。
- 以证券代码唯一匹配已有个股研报；公司名链接遵循“HTML 优先、Markdown 回退、无研报不链接”，不得仅凭简称或模糊名称猜测目标。
- 同一单元格包含多家公司时逐家公司链接；写入或导出前验证每个显式目标真实存在。
- 研报链接只提供企业研究入口，不把产业报告改写成个股推荐；没有研报的公司保持普通文本。

### 7.2 产业链拆分粒度强制口径

上游、中游、下游是导航层级，不是每层一行的压缩配额。同一层级可以重复多行；一个表格行只表达一个可比较子环节。导航层级用于表示资源和产品向终端需求传导的方向，子环节才是承载价值量、技术壁垒、竞争格局、主要参与者和证据的分析单元。

候选内容在以下任一维度存在实质差异时，拆成独立子环节：

- 主要产品或服务；
- 技术路线、接口、标准或认证体系；
- 客户、付费主体或采购决策链；
- 所解决的产业约束或瓶颈；
- 价值量、利润池、资本强度或议价能力；
- 竞争格局或代表主体；
- 是否在技术路线、约束条件、价值量、主要增量或最值得跟踪环节中被单独列为关键项。

只有这些维度基本一致，并且合并后仍能准确填写价值量、壁垒、竞争格局和主要参与者时，才允许共用一行。证据不足不能成为合并理由；证据不足的关键子环节仍需独立保留，并标记“待验证”或“未获取到”。不得使用固定行数，也不得无边界穷举。

反例：不得把“AI 芯片、存储、光模块、交换机、液冷供电”这类产品、标准、客户、瓶颈、利润机制和代表主体不同的内容压缩进同一行。应先拆成可比较子环节，再分别判断价值量、壁垒、竞争格局和公司映射。该反例只说明拆分方法，不是智能算力产业的固定分类模板。

## 8. 标准执行流程

# 步骤 1：明确任务边界

确认：

- 分析的是产业、子行业还是主题概念；
- 是否需要拆分子行业；
- 时间范围和地域范围；
- 用户希望解决的决策问题；
- 本次报告的研究重点。

形成任务声明：

```yaml
task_scope:
  industry:
  sub_industries:
  geography:
  time_range:
  data_cutoff:
  research_question:
  research_focus:
```

---

# 步骤 2：加载最新框架

读取主知识库文件，生成本次运行的 `framework_snapshot`。

检查：

- 模块是否完整；
- 模块顺序是否明确；
- 阶段名称是否明确；
- 是否存在废弃模块；
- 是否有最新修订说明；
- 是否有与模板不一致的部分。

---

# 步骤 3：建立证据计划

根据 `framework_snapshot` 中的模块和必答问题，建立证据清单。

每个问题应映射到：

- 所需事实；
- 所需指标；
- 优先数据源；
- 时间要求；
- 可接受的替代证据；
- 数据缺失时的处理方式。

产业结构模块的证据计划还必须按上游、中游、下游建立公司清单，并为每个主体预先指定：产业链子环节、核心产品或任务、上市属性、业务占比或纯度所需证据、订单/收入核验来源和数据截止时间。

建立公司清单前，先生成子环节清单，并逐项记录拆分或合并依据：

```yaml
chain_subsegment_inventory:
  chain_level:
  subsegment:
  product_or_service:
  customer_or_payer:
  constraint_solved:
  technology_or_standard:
  value_and_profit_mechanism:
  representative_participant_types:
  split_or_merge_rationale:
```

只有子环节清单通过第 7.2 节的拆分判据后，才能据此建立产业链公司映射。不得先按“上游 / 中游 / 下游各一行”压缩，再用公司名单弥补全景表的缺失。

示例结构：

```yaml
evidence_plan:
  - module:
    question:
    required_facts:
    required_indicators:
    preferred_sources:
    freshness:
    fallback_sources:
```

不得先搜集大量资料，再强行套入框架。应由当前框架决定需要搜集什么证据。

---

# 步骤 4：收集和核验信息

证据优先级：

1. 国务院、国家统计部门、部委和监管机构；
2. 行业主管部门和行业协会；
3. 公司公告、年报、招股书和交易所问询；
4. 招标、采购、项目审批、环评和建设信息；
5. 权威研究机构；
6. 主流媒体；
7. 专家观点和二手资料；
8. 市场传闻，仅作为待验证信息。

涉及以下内容时，必须核验最新资料：

- 当前政策；
- 当前规划；
- 当前项目主体；
- 最新投资增速；
- 最新产量、库存和价格；
- 最新订单、产能和投产进度；
- 最新行业利润；
- 当前资本市场状态。

---

# 步骤 5：统一数据口径

对每项关键数据记录：

- 指标名称；
- 指标定义；
- 统计范围；
- 时间范围；
- 名义值或实际值；
- 同比或环比；
- 是否受价格影响；
- 是否存在基数效应；
- 是否存在集中交付；
- 是否可与其他数据比较。

禁止直接建立以下等式：

- 行业增加值 = 行业收入；
- 投资增速 = 已形成有效产能；
- 产量增长 = 利润增长；
- 订单增长 = 收入已确认；
- 收入增长 = 现金流改善；
- 集团承担项目 = 旗下上市公司获得订单；
- 产品能力 = 已进入批量供应。

公司映射中还必须区分：

- 指数或ETF权重，与公司目标产业收入占比；
- 集团/院所承担任务，与旗下上市公司取得订单；
- 目标产业收入占比，与更宽泛的航空航天、军工、通信或数字经济收入占比；
- 已有收入、在手订单、技术储备和潜在应用。

---

# 步骤 6：标记信息性质

所有关键材料分为：

- **事实**：有明确来源支持；
- **推断**：由事实推导；
- **假设**：情景分析前提；
- **待验证**：证据不足；
- **冲突信息**：不同可靠来源结论不一致。

对推断必须说明推导路径。

对冲突信息必须：

1. 比较来源权威性；
2. 比较统计口径；
3. 比较数据时间；
4. 保留仍无法消除的分歧；
5. 不得选择性忽略反向证据。

---

# 步骤 7：逐模块执行分析

严格按照 `framework_snapshot.modules.order` 依次执行。

不得在 Skill 中预设模块名称或固定顺序。

每个模块生成内部分析对象：

```yaml
module_analysis:
  module_id:
  module_name:
  current_judgment:
  confidence:
  key_evidence:
  mechanism:
  current_stage:
  next_stage_conditions:
  risks:
  falsification_indicators:
  evidence_gaps:
  relation_to_other_modules:
```

每个模块必须回答知识库中定义的全部必答问题。

执行产业结构模块时，除框架规定的问题外，先生成“产业链全景子环节”对象：

```yaml
chain_panorama_item:
  chain_level:
  subsegment:
  product_or_service:
  value_level:
  technical_barrier:
  competition_pattern:
  representative_participants:
  evidence_status:
```

再以相同的 `chain_level + subsegment` 身份生成“产业链公司映射”对象：

```yaml
company_mapping:
  chain_level: 上游 / 中游 / 下游
  subsegment:
  company_or_entity:
  listing_status_and_ticker:
  specific_position_and_product:
  business_share_or_purity:
  evidence_status:
  evidence_date_and_source:
  boundary_or_gap:
```

公司映射应覆盖主要子环节和代表主体，但不追求无边界穷举。选择标准是能否帮助判断产业瓶颈、价值量、订单流向、利润传导或竞争格局；仅有概念关联且无产品、客户、订单或收入证据的主体不得列为核心公司。

### 产业结构双向覆盖审计

完成产业链全景和公司映射后，执行双向覆盖审计：

1. 技术路线表中的关键路线，能回指产业链全景中的一个或多个明确子环节；
2. 约束条件表中的核心瓶颈，能回指负责解决该瓶颈的子环节；
3. 价值量与利润池表中的重要环节，能回指产业链全景；
4. “主要增量来源”和“最值得跟踪的产业环节”中的项目，能回指产业链全景；
5. 公司映射中的每个主要子环节，能回指产业链全景；
6. 产业链全景中的每个关键子环节，在公司映射中至少有一个可核验主体；没有可靠主体时明确写“未获取到”。

覆盖检查失败时，先补拆子环节或补证据，不得通过删除关键项、改写成宽泛词或把异质环节合并来获得表面一致。

若某个问题缺少证据：

- 不得省略；
- 标记“待验证”；
- 说明缺少什么数据；
- 给出后续监测方式。

# 步骤 8：分析模块间传导

完成单模块分析后，检查各模块之间的传导关系。

重点识别：

- 规划是否已经转化为实际投资；
- 投资是否形成设备订单和在建产能；
- 新产能何时形成有效供给；
- 供给是否快于需求；
- 库存和价格是否验证供需判断；
- 产业景气是否传导到收入、利润和现金流；
- 资本是否流向真实瓶颈；
- 资本市场定价是否领先产业基本面；
- 产业阶段和市场阶段是否错位；
- 行业出清是否带来集中度提升。

形成传导链：

```text
时代与政策方向
→ 规划与项目
→ 资金与资本开支
→ 设备和建设订单
→ 产能与供给
→ 库存与价格
→ 行业收入与利润
→ 竞争格局变化
→ 资本市场定价
```

传导链须根据知识库最新框架动态调整。

---

# 步骤 9：识别矛盾、错位和反证

至少检查：

- 长期方向强，但短期需求弱；
- 规划规模大，但资金未落实；
- 投资高增长，但有效产能尚未形成；
- 需求增长，但供给扩张更快；
- 产量增长，但价格和利润下降；
- 收入增长，但经营现金流恶化；
- 产业仍处早期，市场已进入泡沫；
- 行业整体出清，龙头已经再成长；
- 资本流入增加，但主要用于重复建设；
- 市场关注度高，但实际订单和利润不足。

输出时必须说明这些错位对判断的影响。

---

# 步骤 10：形成综合结论

综合结论至少包括：

- 产业长期方向；
- 当前产业位置；
- 当前核心矛盾；
- 第一约束；
- 主要增量来源；
- 资源和订单流向；
- 下一阶段转换条件；
- 主要风险；
- 反证指标；
- 仍待验证的数据；
- 置信度。

不得只给阶段标签。

每个阶段判断必须附：

- 判断依据；
- 当前证据；
- 下一阶段条件；
- 可能推翻判断的指标。

不得跳过模板中的章节，不得将完整模板压缩为摘要、专项、监测或比较版本。即使用户只关注一个模块，也应在完整报告中提高该模块的分析深度，其他模块仍须保留并据证据完成或标记“待验证”。

# 步骤 13：保存 Markdown 并生成 HTML 阅读版

完整分析和内容质量检查通过后，必须执行以下收尾流程。该步骤属于每次产业分析的必做项，不以用户是否另行提出“保存”或“导出 HTML”为前提。

## 13.1 确定对应目录

按以下优先级确定输出目录：

1. 用户明确指定目录时，使用用户指定目录；
2. 已存在该产业的自动化目录时，复用已有目录，不得创建同义或重复目录；
3. 未指定且没有既有目录时，根据报告“产业属性”中的**主要属性**路由到：
   - 战略资源：`sources/automations/战略资源/<产业名称>/`
   - 支柱产业：`sources/automations/支柱产业/<产业名称>/`
   - 新兴产业：`sources/automations/新兴产业/<产业名称>/`
   - 其他或主属性证据不足：`sources/automations/其他产业/<产业名称>/`
4. 一个产业同时具备多种属性时，只按主要属性选择目录，次要属性写入报告，不复制到多个目录；
5. 目录归类存在会实质改变落点的歧义时，先完成证据判断；仍无法确定时，明确询问用户，不得静默任选目录。

## 13.2 保存 Markdown 母稿

使用以下默认命名：

```text
YYYY-MM-DD-<产业名称>产业完整分析报告.md
```

保存要求：

- 使用 UTF-8；
- 完整保留 `0—7` 章、“通用风险与证据附录”和分析元数据；
- 先保存 Markdown，再生成 HTML，Markdown 是可继续维护的唯一母稿；
- 同目录已有同名文件时，只有在本次任务明确属于该报告更新时才覆盖；否则使用能够区分日期或版本的文件名；
- 保存前验证公司研报的每个显式链接目标真实存在。

## 13.3 生成 HTML 阅读版

HTML 与 Markdown 使用相同文件名主体：

```text
YYYY-MM-DD-<产业名称>产业完整分析报告.html
```

使用产业报告专用渲染器：

```powershell
node .agents/skills/bbxm-industry-analysis/scripts/render-industry-report-html.cjs --input <产业报告.md> --output <产业报告.html> --vault-root <知识库根目录>
```

不得调用要求个股 16 模块的 `bbxm-equity-research` 渲染器，也不得绕过 Markdown 母稿直接手写 HTML。

## 13.4 验证产物

导出后运行：

```powershell
node .agents/skills/bbxm-industry-analysis/scripts/test-render-industry-report-html.cjs
```

同时对本次报告执行定向检查：

- [ ] Markdown 与 HTML 均位于同一个对应产业目录；
- [ ] 两份文件名主体一致；
- [ ] HTML 依次包含完整的 `0—7` 章；
- [ ] HTML 包含“产业链公司映射”“业务占比或纯度”和“证据状态”；
- [ ] HTML 目录可用，章节锚点完整；
- [ ] 内部链接和公司研报链接可打开；
- [ ] HTML 不含未解析的 `[[...]]`；
- [ ] Markdown 与 HTML 均无 Unicode 替换字符（U+FFFD）或典型中文乱码；
- [ ] 未调用个股报告渲染器；
- [ ] 最终回复列出 Markdown 和 HTML 的实际保存路径。

若通用回归测试因与本次报告无关的旧样例或缺失文件失败，仍须对本次报告完成上述定向检查，并在最终回复披露回归测试失败的具体边界；不得把无关失败描述为本次 HTML 已通过全部测试。

## 9. 失败与降级处理

### 9.1 框架文件不可用

停止正式分析，不使用历史框架替代。

### 9.2 框架结构不清晰

优先根据标题和正文提取；若存在多个合理解释且会改变结论，向用户说明歧义。

### 9.3 模板文件不可用

停止生成正式报告，说明无法加载完整产业分析报告模板。不得自行改用简版、专项版或通用摘要替代。

### 9.4 数据不足

输出“阶段性判断”，明确：

- 缺失数据；
- 暂时采用的替代证据；
- 结论置信度；
- 后续应监测的指标。

### 9.5 最新数据尚未发布

使用最近可得数据，并明确数据截止时间，不得将旧数据描述为当前状态。

### 9.6 HTML 导出失败

- 保留已完成且验证通过的 Markdown 母稿；
- 检查渲染器、Node 运行时、报告结构、显式链接和 UTF-8 编码；
- 修复后重新导出并验证，不得用手写 HTML 绕过专用渲染器；
- 若仍无法生成，明确报告 Markdown 路径、失败原因和未完成的 HTML 路径，不得宣称本次产业分析任务已完整完成。
